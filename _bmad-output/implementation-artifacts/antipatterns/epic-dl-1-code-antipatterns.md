## Story dl-1-4 (2026-04-16)

| Severity | Issue | Fix |
|----------|-------|-----|
| dismissed | "AC #5 implementation incomplete: TWDT watchdog initialization logic is missing in `main.cpp`" | FALSE POSITIVE: FALSE POSITIVE. The ESP32 Arduino framework automatically initializes the TWDT at boot with CONFIG_ESP_TASK_WDT_TIMEOUT_S=5 (default configuration). The code comment on lines 976-977 explicitly documents this. The implementation correctly calls `esp_task_wdt_add(NULL)` to enroll the loop task in the already-initialized watchdog. No explicit `esp_task_wdt_init()` call is required or expected in Arduino framework projects. |
| dismissed | "Unchecked WDT enrollment: `esp_task_wdt_add(NULL)` is called without `esp_task_wdt_init()` configuration" | FALSE POSITIVE: Same as above - the TWDT is pre-initialized by the Arduino framework. | dismissed | "Performance: `esp_task_wdt_reset()` is called repeatedly at the top of every `loop()` iteration, causing unnecessary overhead" | FALSE POSITIVE: FALSE POSITIVE. Calling `esp_task_wdt_reset()` at the top of the loop is the CORRECT watchdog pattern for preventing false positives. The additional calls around blocking HTTP fetch operations (lines 1249, 1251) are also correct to prevent timeouts during long network operations. This is not a performance issue - it's proper watchdog supervision implementation. |
| dismissed | "Missing error handling: boot restore doesn't check `onManualSwitch` return value before persisting to NVS" | FALSE POSITIVE: PARTIALLY VALID CONCERN but not actionable. The function signature shows `onManualSwitch()` returns `void`, so checking return values is impossible. The underlying concern about unchecked failures is valid, but the implementation is internally defensive - `onManualSwitch` only updates orchestrator state if `ModeRegistry::requestSwitch()` succeeds. If the switch fails, the orchestrator retains its previous state. While there's a theoretical risk of NVS/orchestrator mismatch, the current implementation matches the architecture and no fix is specified in the story scope. |
