## 🐛 Correctness & Safety

- **🐛 Bug:** Calling `Update.abort()` after `Update.end(true)` returns false is architecturally invalid. Once `Update.end()` is called (success or failure), the update session is finalized. `abort()` is only valid during active write operations, not after finalization. This could leave the partition in an undefined state.
  - 📍 `firmware/adapters/WebPortal.cpp:519-520`
  - 🔄 Reproduction: Upload corrupted firmware that fails hash verification; `Update.end(true)` returns false, then `Update.abort()` is called on already-finalized session.

- **🐛 Bug:** No validation that uploaded file size ≤ partition size before `Update.begin()`. If a 2MB file is uploaded to a 1.5MB partition, flash corruption occurs.
  - 📍 `firmware/adapters/WebPortal.cpp:483-491`
  - 🔄 Reproduction: Craft POST with Content-Length > 1.5MB and stream data; partition overflow occurs silently.

- **🔒 [HIGH] Security:** No rate limiting or authentication on `/api/ota/upload` - anyone on the network can flash arbitrary firmware (magic byte 0xE9 is trivial to forge).
  - 📍 `firmware/adapters/WebPortal.cpp:399-531`
  - ⚠️ Impact: Complete device compromise by anyone with network access

- **🔒 [MEDIUM] Security:** Missing Content-Length validation before processing - attacker could exhaust flash by sending infinite stream.
  - 📍 `firmware/adapters/WebPortal.cpp:440-442`
  - ⚠️ Impact: Denial of service via flash exhaustion

**File:** `firmware/adapters/WebPortal.cpp:517-527`

**Issue:** `Update.abort()` after failed `Update.end()` is undefined behavior.

```cpp
// WRONG:
if (!Update.end(true)) {
    Update.abort();  // REMOVE THIS LINE
    state->valid = false;
    // ...

// CORRECT:

    state->valid = false;
    state->error = "Firmware verification failed";
    state->errorCode = "VERIFY_FAILED";

    return;
```
