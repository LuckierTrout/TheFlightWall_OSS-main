## 🏛️ Architectural Sins

- **[6/10] Single Responsibility:** The `OTAUploadState` struct includes `AsyncWebServerRequest* request`. This couples the state directly to the web server request, which might lead to issues if the request object's lifecycle is different or if other parts of the system need to track OTA state independently of a specific request. This violates the Single Responsibility Principle as the state struct is now responsible for both holding OTA state and referencing a specific request.
  - 📍 `firmware/adapters/WebPortal.cpp:83`
  - 💡 Fix: Decouple the `OTAUploadState` from `AsyncWebServerRequest` by using a unique identifier for the upload session (e.g., a `uint32_t` session ID) generated at the start of the upload and passed with the `request->onDisconnect` callback. The `findOTAUpload` helper would then search by this ID.

## 🔧 Maintainability Issues

- **💣 Tech Debt:** The success block for `POST /api/ota/upload` includes repeated initialization logic for `esp_timer_handle_t otaRebootTimer`. While `esp_timer_create_args_t args = {};` and `esp_timer_create(&args, &otaRebootTimer);` are guarded by `if (!otaRebootTimer)`, this pattern is verbose and introduces potential for subtle bugs if the timer handle is accidentally reset or if the `create` function were to fail, which is not handled. This is a copy-paste from `_handlePostReboot` and `_handlePostReset`.
  - 📍 `firmware/adapters/WebPortal.cpp:520`
  - 💥 Explosion radius: Increased code verbosity, potential for slight heap fragmentation if `esp_timer_create` is called multiple times due to a bug, and violation of DRY (Don't Repeat Yourself). Better to have a single `setup` function for the timer.
