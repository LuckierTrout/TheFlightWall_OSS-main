## Findings

1. `IMPORTANT` AC #3 is still incomplete. `.ota-upload-zone.drag-over` changes only `border-color`; the base `2px dashed` style remains, so the user never gets the required solid drag-over affordance. This is also a task-completion lie because Task 6 explicitly marked the solid border as done. `firmware/data-src/style.css:462`

2. `IMPORTANT` AC #11 is not enforced in CSS. `Upload Firmware`, `Cancel`, and `Download Settings` rely on padding only; there is no `min-height: 44px`/`min-width: 44px`. `btn-ota-cancel` is especially undersized on default mobile metrics. `firmware/data-src/style.css:174`, `firmware/data-src/style.css:487`

3. `IMPORTANT` `startRebootPolling()` can stack requests on a device that is explicitly resource-constrained. Because it uses `setInterval`, a slow or half-open `/api/status` call does not block the next poll. During reboot/Wi-Fi recovery, this can create multiple concurrent status fetches against the same ESP32. `firmware/data-src/dashboard.js:2818`

4. `IMPORTANT` Page initialization now sends `/api/settings`, `/api/status`, `/api/display/modes`, `/api/logos`, and `/api/layout` immediately and in parallel. That is the opposite of the architecture note to keep concurrent clients around 2-3, and it makes the “dashboard loads within 3 seconds” AC harder to satisfy after this story, not easier. `firmware/data-src/dashboard.js:2866`

5. `IMPORTANT` The OTA success path treats rollback-ack reset as best-effort but never checks whether it actually persisted. If `Preferences::begin()` or `putUChar("ota_rb_ack", 0)` fails, the update still reports success and a later rollback can remain permanently hidden. `firmware/adapters/WebPortal.cpp:507`

6. `IMPORTANT` This story added a stateful OTA UI and new persistence behavior, but there is no new automated coverage for file validation, reboot polling, rollback acknowledgment, or settings export. That leaves the highest-risk logic entirely review/manual-test dependent. `firmware/data-src/dashboard.js:2666`, `firmware/adapters/WebPortal.cpp:532`

7. `MINOR` `loadFirmwareStatus()` silently ignores both non-OK responses and network failures. If `/api/status` breaks on first load, the firmware version stays empty and the rollback banner never appears, with no toast and no retry path. `firmware/data-src/dashboard.js:2631`

---

## 🛠️ Suggested Fixes

1. Add `border-style: solid` to `.ota-upload-zone.drag-over`.
2. Encode touch-target guarantees in CSS with explicit `min-height: 44px` and `min-width: 44px` for the new OTA/System controls.
3. Replace reboot `setInterval` polling with a single-flight recursive `setTimeout`, or gate each poll behind an `inFlight` flag/abortable request.
4. Sequence or lazy-load dashboard startup requests so essential status/settings render first and non-critical data follows.
5. Check and log `Preferences` open/write failures when clearing `ota_rb_ack`; do not silently claim success if the rollback-banner invariant was not preserved.
6. Add codebase tests or smoke coverage for OTA file validation, reboot polling timeout/success paths, rollback acknowledgment persistence, and export endpoint behavior.
