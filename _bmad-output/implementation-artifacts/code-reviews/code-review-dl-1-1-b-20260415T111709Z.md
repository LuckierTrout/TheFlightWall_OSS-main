## 🏛️ Architectural Sins

- **[8/10] Abstraction / Rule 29 Violation:** Architecture Rule 29 explicitly mandates: "Mode settings schemas are declared as `static const` in the mode's own `.h` file". The developer circumvented this by using `extern const` in the header and defining the schema in the `.cpp` file, claiming it was to avoid an ODR violation. While the compiler complaint was real, arbitrarily breaking a core architectural enforcement rule without establishing a new pattern is a failure.
  - 📍 `firmware/modes/ClockMode.h:16`

- **[4/10] Abstraction / Rule 23 Violation:** Rule 23 explicitly states that `MEMORY_REQUIREMENT` must reflect "worst-case heap from `init()`, not object size". The code literally contains a comment calculating the object size (`vtable ptr + _lastRenderedSecond + _format = ~6 bytes`) to justify setting the requirement to 64. Since `init()` allocates 0 bytes on the heap, this completely misunderstands the rule's instruction.
  - 📍 `firmware/modes/ClockMode.h:23`

## 🐛 Correctness & Safety

- **🐛 Bug / Architecture Flaw:** Broken Hot Reload for Mode Settings. Architecture Rule 18 mandates that modes cache their settings during `init()`. However, `ClockMode` caches `_format` in `init()` and never reads it again. When `setModeSetting` is called and fires `g_configChanged`, the display task rebuilds the `RenderContext` but *does not* re-initialize the active mode. Furthermore, `ModeRegistry::requestSwitch()` ignores requests to switch to the already-active mode. Thus, a setting change submitted via API will NEVER take effect until the device reboots or the mode is switched away and back.
  - 📍 `firmware/modes/ClockMode.cpp:42`
  - 🔄 Reproduction: Call `POST /api/display/mode` with `{"mode": "clock", "settings": {"format": 1}}` while Clock Mode is active. The NVS value will update, but the display will continue showing the old format.
