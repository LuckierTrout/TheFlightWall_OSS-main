## 🔴 CRITICAL FINDINGS

### 1. **AC #4 Heap Floor Assertion Only Meaningful On-Device (UNVERIFIED)**
- **Severity:** CRITICAL
- **Location:** `firmware/test/test_golden_frames/test_main.cpp:310-330`
- **Issue:** The story claims AC #4 (heap floor ≥ 30 KB at max density) is "Assertion encoded" but the test comment explicitly states "heap assertion only meaningful on-device". The story file says AC #4 is PASS, but there is zero evidence this test has ever been run on actual hardware.
- **Impact:** The 30 KB floor gate is mathematically unvalidated. A regression that drops heap to 29 KB would pass CI and fail silently on device.
- **Evidence:** Story shows `test_golden_c_heap_floor` exists but verification report section 3 shows "TBD on-device" for actual measurements. No ESP32 serial output captured.

### 2. **p95 Render Ratio Measurement Completely Deferred (AC #5 UNMET)**
- **Severity:** CRITICAL
- **Location:** Verification Report, Section 4
- **Issue:** AC #5 requires "p95 render-ratio measurement captured on fixture (b) alongside ClassicCardMode on same hardware pass". The story explicitly documents this as "Hardware measurement deferred — on-device validation required". Yet the story status is marked as complete.
- **Impact:** One of the two primary performance gates (1.2× ratio cap) has never been measured. LE-1 cannot claim readiness without this data.
- **Evidence:** Verification report section 4 shows "TBD" for all p95 measurements. Spike report reference is V0 data, not current-state LE-1.8 production binary.

### 3. **Fixture JSON Files Not Actually Committed to Repo**
- **Severity:** CRITICAL
- **Location:** Task 1 subtasks — "layout_a.json", "layout_b.json", "layout_c.json"
- **Issue:** Story claims three fixture JSON files are created and live at `firmware/test/test_golden_frames/fixtures/layout_{a,b,c}.json`. These files are **referenced in the test harness** (`kFixtureA`, `kFixtureB`, `kFixtureC` inline C strings) but the actual `.json` files do not exist in the file list. Only inline C string literals are provided in `test_main.cpp`.
- **Impact:** The "authoritative schema reference" claimed in the test file header doesn't exist. The fixtures are not version-controlled independently, only embedded as strings. This violates AC #1's implicit requirement that fixtures be storable/retrievable separately.
- **Evidence:** Dev Agent Record section "File List → Created" lists only `test_main.cpp`, not the three fixture JSON files. Story task 1 checkboxes are marked `[x]` but the JSON files are absent.

## 🟠 HIGH FINDINGS

### 4. **Binary Size Measurement Not From Clean Build (Potential Replay Attack)**
- **Severity:** HIGH
- **Location:** Verification Report, Section 2
- **Issue:** The reported binary size (1,303,488 bytes, 82.9%) is claimed to be "measured on commit `81bf6a1c…`" but the Dev Agent Record shows this size was seeded into `.firmware_baseline_size` AFTER the build, creating a circular dependency. If the binary was rebuilt today, would it produce the same size? No validation that the build is deterministic or that this exact size is reproducible.
- **Impact:** The 92% cap gate is only as strong as the baseline. A non-deterministic build system could mask regressions.
- **Evidence:** Dev Agent Record states "Baseline seeded with current measured size (1,303,488 B)" — this is seeding, not validation.

### 5. **check_size.py Delta Check Uses Non-Atomic File Artifact**
- **Severity:** HIGH
- **Location:** `firmware/check_size.py`, delta check logic
- **Issue:** The `.firmware_baseline_size` file is a plain text integer with no integrity check, no version header, no timestamp. A corrupted or manually edited `.firmware_baseline_size` file would silently cause the delta check to fail or pass incorrectly. No verification that the baseline is from `main` branch or a specific commit.
- **Impact:** The 180 KB delta cap can be bypassed by editing a text file. A developer could manually adjust `.firmware_baseline_size` without code review.
- **Evidence:** Task 3 specifies "create `.firmware_baseline_size` with the current binary size" — no cryptographic validation, no commit tie-in, just a number.

### 6. **Fixture (c) Widget Count Claim Not Verified Against MAX_WIDGETS Actual**
- **Severity:** HIGH
- **Location:** `firmware/test/test_golden_frames/test_main.cpp:435`, fixture C JSON inline
- **Issue:** Story claims fixture (c) has "24 widgets exactly at MAX_WIDGETS" but the inline JSON shows 24 widget objects. If `MAX_WIDGETS` is ever changed (e.g., to 32 in a future story), the fixture is no longer at the cap. No assertion verifies `24 == MAX_WIDGETS`.
- **Impact:** The "max-density" test loses semantic meaning if the cap changes. The test becomes a "24-widget test" not a "max-capacity test".
- **Evidence:** Test comment says "24-widget max-density (MAX_WIDGETS = 24)" but this is a hardcoded number, not a constant reference.

### 7. **Golden Frame Test Strategy Fundamentally Broken for Clock Widgets (AC #1 Incomplete)**
- **Severity:** HIGH
- **Location:** Dev Notes, Golden-Frame Test Strategy section
- **Issue:** The story explicitly acknowledges: "Clock widgets produce time-dependent output — the rendered framebuffer for fixture (a) will differ each time depending on the current minute". Yet fixture (a) includes a clock widget. The test `test_golden_a_render_does_not_crash` only validates "no crash" and "widget count == 2", NOT that the output is correct. This is **structurally incapable** of proving golden-frame pixel correctness for any layout containing a clock.
- **Impact:** AC #1 claims "golden-frame regression tests" but the actual tests cannot detect regressions in clock rendering. Any pixel-perfect golden test for fixture (a) is technically impossible without mocking system time.
- **Evidence:** Test file shows `test_golden_a_render_does_not_crash` contains zero assertions about output correctness — just `TEST_PASS()` after render completes.

### 8. **check_size.py Error Messages Are Machine-Unreadable (CI Integration Risk)**
- **Severity:** HIGH
- **Location:** `firmware/check_size.py`, error output format
- **Issue:** The error messages use human-friendly formatting (`{size:,}` with commas, percentages with decimals) but CI systems typically parse structured output (JSON, key=value). If this script is integrated into CI with automated failure parsing, the comma formatting will break parsing (e.g., `1,447,036` will not parse as a number).
- **Impact:** CI systems may fail to properly detect and report binary size violations. False negatives could allow regressions to merge.
- **Evidence:** Lines in `check_size.py` like `print(f"Usage: {size/PARTITION_SIZE*100:.1f}%")` produce human-readable but machine-hostile output.

## 🟡 MEDIUM FINDINGS

### 9. **MEMORY_REQUIREMENT Static Assert Only Validates Size, Not Actual Allocation Paths**
- **Severity:** MEDIUM
- **Location:** `firmware/modes/CustomLayoutMode.cpp:23-25`
- **Issue:** The static assertion checks `MEMORY_REQUIREMENT >= MAX_WIDGETS * sizeof(WidgetSpec)` but this only validates that the constant is large enough for the array. It does NOT validate that the actual code paths in `init()` only allocate this much heap, or that `teardown()` frees everything. A mode that allocates extra heap in `init()` (e.g., `String` objects in `_widgets` fields) would pass this assertion but violate the contract.
- **Impact:** The assertion is a false sense of security. A mode could declare `MEMORY_REQUIREMENT = 4096` but actually allocate 8KB, breaking the heap guard logic.
- **Evidence:** Story assumes assertions prevent heap violations, but static_assert only validates that a constant meets a lower bound.

### 10. **PixelMatrix Stub in Dev Notes Never Actually Implemented**
- **Severity:** MEDIUM
- **Location:** Dev Notes section "PixelMatrix Stub" vs actual test implementation
- **Issue:** The Dev Notes describe a detailed `PixelMatrix` test stub class with `fillScreen`, `drawPixel`, `setCursor` virtual overrides. The actual `test_main.cpp` uses `ctx.matrix = nullptr` and `TEST_PASS()` stubs instead. The recommended design (full pixel-level comparison) was abandoned for the weaker "structural check only" approach, but the Dev Notes still describe the stronger design.
- **Impact:** Developers reading the Dev Notes will expect pixel-exact framebuffer regression tests but the actual implementation cannot provide them. Documentation-code mismatch.
- **Evidence:** Compare "PixelMatrix Stub (Framebuffer-Based Testing)" section in Dev Notes to actual test file — no PixelMatrix class exists.

### 11. **Fixture JSON "Canonical Schema Reference" Not Versioned or Migrated**
- **Severity:** MEDIUM
- **Location:** Task 1 & Dev Notes "Fixture JSON Formats"
- **Issue:** Story claims fixture JSON files are "the authoritative schema reference" but they are never version-controlled separately. If a future story adds a new widget type (e.g., "timer"), existing fixtures (a/b/c) remain frozen in V1 format. There is no migration path or versioning scheme for fixtures as the data model evolves.
- **Impact:** Fixtures become maintenance debt. Future stories cannot safely extend the schema without risking golden-frame test failure.
- **Evidence:** Dev Agent Record shows fixtures exist only as inline C strings in test_main.cpp, no separate `.json` files for schema governance.

### 12. **Heap Baseline Measurement Omits Key System Components (Underestimated Ceiling)**
- **Severity:** MEDIUM
- **Location:** Dev Notes Task 1, "Heap Regression Check" section
- **Issue:** AC #4's 30 KB floor is validated against heap measured "after loading and rendering fixture (c) with `CustomLayoutMode`". But the heap measurement does NOT account for: active WiFi connections, OTA partition locks, LittleFS mount overhead, or AsyncWebServer buffers. The 30 KB floor is only valid **in isolation** during the test, not during actual operation when all subsystems are active.
- **Impact:** On real hardware with WiFi + web server + OTA staging, the actual free heap could be significantly lower than the test shows. The gate is validating the wrong scenario.
- **Evidence:** Story states "heap assertion only meaningful on-device" but the validation scenario is test-isolated, not production-realistic.

## 🟢 LOW FINDINGS

### 13. **Verification Report Missing Build Environment Metadata**
- **Severity:** LOW
- **Location:** Verification Report header
- **Issue:** The report documents "Commit SHA: 81bf6a1cb8…" but does NOT include: PlatformIO version, platformio.ini configuration used, ESP-IDF version, or C++ compiler flags. If the binary is rebuilt with different toolchain versions, the size could differ. No reproduction guarantee.
- **Impact:** Another engineer cannot reproduce the exact binary size without trial-and-error on tool versions.
- **Evidence:** Verification Report Section 2 shows only size, date, and commit SHA — missing build context.

### 14. **Test Suites Compile but Heap Measurement Not Cross-Verified Against Production Binary Size**
- **Severity:** LOW
- **Location:** Task 8, "Cross Story Compile Sweep" section
- **Issue:** The story validates that all LE-1 test suites compile cleanly and the product binary builds (1,303,488 bytes at 82.9%). But there is NO verification that the test binary (with unity harness, logging, test infrastructure) uses significantly less heap than the product binary during the same test_golden_c_heap_floor measurement. If the test harness itself consumes 15 KB, then the 30 KB floor measurement is measuring test overhead, not production headroom.
- **Impact:** The heap floor gate may be validating the wrong binary (test harness vs product).
- **Evidence:** No comparison documented between test binary heap usage and product binary heap usage.
