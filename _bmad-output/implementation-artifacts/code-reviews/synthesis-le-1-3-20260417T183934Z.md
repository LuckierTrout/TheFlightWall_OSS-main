### High

### Medium

- **Issue**: Misleading `_activeModeIndex` write in `requestForceReload()` races with Core 0 reading it between the two stores. Analysis: both `_activeModeIndex` and `_requestedIndex` are `std::atomic<uint8_t>`, and Core 0 only reads `_activeModeIndex` *after* it has already consumed and cleared `_requestedIndex`. The narrow window where Core 0 could observe `_activeModeIndex == MODE_INDEX_NONE` without a corresponding pending request is benign — it would simply render a tick with no active mode (same as startup). This is acceptable for an infrequent layout-reload path. Documented in the implementation comment. | **Source**: Synthesis analysis | **File**: `firmware/core/ModeRegistry.cpp` | **Fix**: Documented the race window and its benign nature in the implementation comments. No code change needed.

The spike build environment is an intentional parallel artifact with zero impact on `esp32dev`. Not a documentation lie.

- **Claimed Issue**: AC #7 hook is "too narrow" — only triggers for `custom_layout` active mode | **Raised by**: Reviewer C | **Dismissal Reason**: AC #7 explicitly specifies *"if `CustomLayoutMode` is the active mode"*. The concern about layout changes while in another mode is a LE-1.4 concern (when the activate endpoint is implemented). The current hook exactly matches AC #7's scope.

- **Claimed Issue**: `render()` accesses stale `RenderContext` | **Raised by**: Reviewer A | **Dismissal Reason**: `render(ctx, flights)` receives `ctx` as a `const RenderContext&` parameter passed fresh every frame from `ModeRegistry::tick()` → `_activeMode->render(ctx, flights)`. There is no caching of `ctx` members in `CustomLayoutMode`. The concern is hypothetical.

- **Claimed Issue**: OCP violation in `main.cpp` mode table | **Raised by**: Reviewer A | **Dismissal Reason**: The static `MODE_TABLE` is the project's intentional architecture (AR5: "Adding a future mode = new modes/*.h/.cpp + one ModeEntry line here"). This is a design choice documented in `main.cpp`, not a violation.

- **Claimed Issue**: Widget dimension bounds not validated in `parseFromJson()` | **Raised by**: Reviewer C | **Dismissal Reason**: Widget renderers (TextWidget, ClockWidget, LogoWidget) are responsible for clipping to their spec bounds and the matrix dimensions. The GFX library clips all drawing operations to the matrix dimensions natively. This is a defense-in-depth concern, not a correctness bug. Out of LE-1.3 scope.

- **Claimed Issue**: `JsonDocument` has no explicit capacity — could spike higher than `MEMORY_REQUIREMENT` | **Raised by**: Reviewer C | **Dismissal Reason**: ArduinoJson v7's `JsonDocument` uses dynamic sizing from the heap pool. The 4096-byte `MEMORY_REQUIREMENT` is evaluated pre-init by `ModeRegistry`'s heap guard (`freeHeap < MEMORY_REQUIREMENT + 30KB`). The 30KB margin above `MEMORY_REQUIREMENT` provides substantial headroom for transient JSON allocation. The story Dev Notes document this explicitly.

- **Claimed Issue**: `parseHexColor()` redundant validation loop before `strtol` | **Raised by**: Reviewer C | **Dismissal Reason**: The loop validates character-by-character first so `strtol` is only called with known-good input. The `endptr` check is then a belt-and-suspenders guard. Not redundant — defensive layering. No bug.

- **Claimed Issue**: Missing null check on `LayoutStore::getActiveId()` return | **Raised by**: Reviewer C | **Dismissal Reason**: `LayoutStore::getActiveId()` returns `String` (not `const char*`). An empty `String` has a valid `.c_str()` returning `""`. `LayoutStore::load("", out)` is handled — the empty-string path falls through to the default layout. No null dereference possible.

- **Claimed Issue**: AC #1 "5th ModeEntry" language is ambiguous | **Raised by**: Reviewer C | **Dismissal Reason**: The mode table has 5 entries (indices 0–4); `custom_layout` is at index 4. "5th ModeEntry" means ordinal 5 = index 4. The implementation is unambiguously correct.

- **Claimed Issue**: `check_size.py` not provided for verification | **Raised by**: Reviewer C | **Dismissal Reason**: `check_size.py` is referenced in `platformio.ini:19` as `extra_scripts = pre:check_size.py` and its output is visible in every build: `"Binary size: 1,283,520 bytes ...
