That means the shipped firmware checks the wrong releases feed, so AC #8 / Task 3 are not actually satisfied.

2. **🟠 AC #9 is still not met: there are no native host tests for the pure helper logic.** The story explicitly asks for native tests where feasible ([story](/Users/christianlee/App-Development/TheFlightWall_OSS-main/_bmad-output/implementation-artifacts/stories/dl-6-1-ota-version-check-against-github-releases.md:35)), but [platformio.ini](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/platformio.ini:1) defines only `env:esp32dev`, and the test file exercises host-testable helpers like `normalizeVersion`, `compareVersions`, and JSON parsing while still requiring device-side Unity ([test_main.cpp](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/test/test_ota_updater/test_main.cpp:16)). The story’s own debug notes admit the tests require physical hardware.

3. **🟠 The claimed “bounded heap” fix is incomplete; large release notes can still blow heap.** [OTAUpdater.cpp](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/core/OTAUpdater.cpp:172), [OTAUpdater.cpp](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/core/OTAUpdater.cpp:250). Both parse paths deserialize into dynamic `JsonDocument` and keep the full `body` string in memory before truncating it to 512 bytes ([OTAUpdater.cpp](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/core/OTAUpdater.cpp:200), [OTAUpdater.cpp](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/core/OTAUpdater.cpp:278)). A release with verbose notes or many assets can still cause exactly the heap pressure the review follow-up claims was fixed.

4. **🟠 OTAUpdater is not concurrency-safe, and the current integration already makes that dangerous.** The module exposes raw mutable buffers for `lastError`, version, notes, and asset URLs ([OTAUpdater.h](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/core/OTAUpdater.h:64), [OTAUpdater.cpp](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/core/OTAUpdater.cpp:31)). `checkForUpdate()` clears and rewrites those buffers in place ([OTAUpdater.cpp](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/core/OTAUpdater.cpp:329)), while later consumers read them from async handlers ([WebPortal.cpp](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/adapters/WebPortal.cpp:1454), [WebPortal.cpp](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/adapters/WebPortal.cpp:1533)) and the download task uses the same URLs after spawn ([OTAUpdater.cpp](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/core/OTAUpdater.cpp:582), [OTAUpdater.cpp](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/core/OTAUpdater.cpp:668)). Two clients hitting check/status/pull concurrently can tear JSON responses or poison an in-flight OTA.

5. **🟡 The “tests strengthened” claim is overstated; some assertions are still weak enough to miss real regressions.** [test_main.cpp](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/test/test_ota_updater/test_main.cpp:140), [test_main.cpp](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/test/test_ota_updater/test_main.cpp:450). `test_parse_valid_release()` only checks that the binary URL contains `"firmware.bin"`, which would still pass if the parser accidentally stored the `.sha256` URL.
