## Issues Dismissed

- **Claimed Issue**: State corruption in `onManualSwitch` — state updated before `requestSwitch` validation.
  **Raised by:** Reviewer A (Critical), also referenced in ANTIPATTERNS doc.
  **Dismissal Reason:** FALSE POSITIVE. `ModeOrchestrator.cpp:83` calls `requestSwitch()` first; line 85 returns early on failure. State updates happen only at lines 88–97, strictly after successful validation. The ANTIPATTERNS doc describes the **dl-1.2** bug; the dl-1.3 implementation correctly fixed it. Reviewer A appears to have projected the expected antipattern onto the current code without reading the actual implementation order.

- **Claimed Issue**: Architectural Rule 24 violation in `main.cpp` boot path — `requestSwitch` called outside the orchestrator.
  **Raised by:** Reviewer A (Critical), Reviewer B (implicit support).
  **Dismissal Reason:** FALSE POSITIVE. `grep -rn "ModeRegistry::requestSwitch" firmware/ --exclude-dir=test` returns **only** `ModeOrchestrator.cpp` (5 call sites) and `ModeRegistry.cpp` (definition). Zero hits in `main.cpp`, `WebPortal.cpp`, or any other production file. The boot path correctly calls `ModeOrchestrator::onManualSwitch()`.

- **Claimed Issue**: "Lying Task Claim" — dev notes falsely claim Rule 24 compliance.
  **Raised by:** Reviewer A.
  **Dismissal Reason:** FALSE POSITIVE. The dev notes are accurate. See grep audit above.

- **Claimed Issue**: `test_manual_switch_unknown_mode_requestswitch_fails` omits `getManualModeId()` assertion to hide state corruption.
  **Raised by:** Reviewer A (ANTIPATTERNS doc, dl-1.2 fix applied).
  **Dismissal Reason:** FALSE POSITIVE. The assertion `TEST_ASSERT_EQUAL_STRING("classic_card", ModeOrchestrator::getManualModeId())` IS present at test line 320, with an explicit comment "regression guard for line 81 bug." - **Claimed Issue**: Boot path NVS double-write "race condition" (Critical).
  **Raised by:** Reviewer B.
  **Dismissal Reason:** NOT A RACE CONDITION. Both the direct write (`ConfigManager::setDisplayMode`) and the later debounced write (`tickNvsPersist`) write the **same value** in deterministic order. There is no conflicting write, no corrupted final state. The direct writes occur in `setup()` before any FreeRTOS tasks start, providing boot-time durability. Severity is Low (documentation gap), not Critical.

- **Claimed Issue**: POST response shows "stale" active mode — violates AC #1 convergence.
  **Raised by:** Reviewer B.
  **Dismissal Reason:** FALSE POSITIVE — by design. AC #1 states the active mode "converges with orchestrator intent **after `ModeRegistry::tick`**." The POST response correctly returns `switching_to` (the new mode) and `active` (pre-tick current mode). The design explicitly requires clients to poll `GET /api/display/modes` for convergence. Reviewer B's "fix" was purely a comment addition with no behavior change.

- **Claimed Issue**: `findModeName` lambda in `main.cpp` duplicates registry table iteration (SRP / logic duplication).
  **Raised by:** Reviewer A (Low), Reviewer B (implicit).
  **Dismissal Reason:** ALREADY DEFERRED. This was identified and explicitly deferred in the **dl-1.2 synthesis** (ANTIPATTERNS: "Deferred — Would require adding `ModeRegistry::getModeName()` helper, which exceeds synthesis scope"). Tracked as a future improvement.

- **Claimed Issue**: Missing NVS persistence test for boot restore paths.

  **Dismissal Reason:** FALSE POSITIVE. `test_wdt_recovery_persists_clock_to_nvs` at line 407–416 directly asserts `ConfigManager::getDisplayMode() == "clock"` after WDT recovery boot restore. Coverage exists.

- **Claimed Issue**: "Scope creep" — dl-1.4 WDT recovery in dl-1.3 scope.
