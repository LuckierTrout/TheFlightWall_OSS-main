## Suggested Fixes

### 1. Decouple rollback reporting from mark-valid success

**Files:** `firmware/src/main.cpp`  
**Issue:** rollback detection and rollback-status publication are incorrectly tied to the WiFi/timeout self-check success path.

**Fix:** publish the rollback `SystemStatus` warning once immediately after `SystemStatus::init()` becomes available, or on the first loop tick regardless of WiFi/timeout and regardless of `esp_ota_mark_app_valid_cancel_rollback()` result.

### 2. Fix the boot-time measurement and pending-verify probe

**Files:** `firmware/src/main.cpp`  
**Issue:** timing starts late, and OTA state probe failures are silently treated as “already valid”.

**Fix:** assign `g_bootStartMs` before `delay(200)`, and keep `s_isPendingVerify` in an “unknown” state when `esp_ota_get_state_partition()` fails so the code can retry or emit an explicit OTA error.

### 3. Add retry throttling

**Files:** `firmware/src/main.cpp`  
**Issue:** persistent `esp_ota_mark_app_valid_cancel_rollback()` failures hammer the loop.

**Fix:** add a retry interval such as 1s and suppress duplicate error logs/status writes unless the error code changes.

### 4. Add tests before calling this done

**Files:** `firmware/test/...`  
**Issue:** the story currently relies on reasoning and manual device testing for rollback-critical behavior.

**Fix:** add unit or seam-based tests for:
- pending-verify + WiFi connected
- pending-verify + 60s timeout
- normal boot with no OTA status emission
- rollback-detected boot serialization in `/api/status`
