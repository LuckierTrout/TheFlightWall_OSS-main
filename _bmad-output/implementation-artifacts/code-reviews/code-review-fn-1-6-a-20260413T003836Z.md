## ⚡ Performance &amp; Scalability

- **[CRITICAL] Lack of early server-side file size validation:**
  - 📍 `firmware/adapters/WebPortal.cpp:468`
  - The server-side OTA upload handler in `WebPortal.cpp` only validates the magic byte upfront. It does not check the incoming file's `Content-Length` header against the OTA partition size before starting the stream to flash. This can lead to the ESP32 spending significant time and resources processing a large, invalid file that will eventually fail, reducing efficiency and potentially leading to a denial-of-service if large files are frequently sent.
  - 💡 Fix: Add a check `if (request->contentLength() > partition->size)` at the beginning of the `index == 0` block in the OTA upload handler to immediately reject oversized files.

  - 🔄 Reproduction: In `startRebootPolling()`, if the final `FW.get('/api/status')` succeeds very close to the 60-second timeout, it's possible for both the "Device unreachable" message (from the timeout logic) and the "Updated to v[new version]" toast (from the successful poll) to display. - **🔒 [CRITICAL] Security:** Lack of explicit server-side file size validation in OTA upload.
  - 📍 `firmware/adapters/WebPortal.cpp:468`
  - ⚠️ Impact: Malicious actors could bypass client-side JavaScript validation and send firmware files larger than the allotted OTA partition. The ESP32 would then proceed to stream these oversized files, consuming memory and CPU cycles while writing to flash, only to fail reactively when the partition limit is hit or `Update.end()` is called. This could lead to resource exhaustion and a denial-of-service attack on the device.
