## 🏛️ Architectural Sins

- **Dependency Inversion:** The `OTAUpdater` hardcodes dependencies on `WiFiClientSecure` and `HTTPClient`, making it impossible to test the API logic without bringing up the full network stack or mocking at the class level.
  - 📍 `firmware/core/OTAUpdater.cpp:165`
  - 💡 Fix: Inject a `Client` or `HttpClient` interface to decouple the business logic from the hardware network stack.

---

## ⚡ Performance &amp; Scalability

- **[MEDIUM] Unbounded memory risk:** While stream parsing is used, the `_releaseNotes` buffer is fixed-size (`NOTES_MAX_LEN = 512`), but if the JSON structure contains massive fields, the ArduinoJson parser might still allocate a large document if not perfectly tuned.
  - 📍 `firmware/core/OTAUpdater.cpp:249`
  - 💡 Fix: Further constrain the `JsonDocument` filter and use a strict `deserializeJson` limit.

---

## 🐛 Correctness &amp; Safety

- **🐛 Bug:** `_resetForTest()` does not clear all state variables thoroughly, specifically the handles for tasks which could remain dangling if a task was spawned and not joined.
  - 📍 `firmware/core/OTAUpdater.cpp:94`
  - 🔄 Reproduction: Call `init()`, then `_resetForTest()`, then check `_downloadTaskHandle` (should be null, but was not explicitly set to null in reset for all cases if task wasn't fully cleaned).

- **🔒 [MEDIUM] Security:** `client.setInsecure()` is used for all GitHub API connections. This disables SSL certificate verification, making the update process susceptible to Man-in-the-Middle (MITM) attacks.
  - 📍 `firmware/core/OTAUpdater.cpp:165`
  - ⚠️ Impact: An attacker could inject a malicious binary URL or version tag.
