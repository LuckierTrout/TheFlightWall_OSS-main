## Issues Verified (by severity)

### Critical

- **Cross-core race condition: `_otaMode` not atomic**
  - **Source**: Validator B
  - **File**: `firmware/core/ModeRegistry.h:115`, `firmware/core/ModeRegistry.cpp:32`
  - **Evidence**: `_otaMode` declared as `bool` but written on Core 1 (OTA task/upload handler) and read on Core 0 (display task). No synchronization guarantees visibility.
  - **Impact**: Display task may miss OTA mode flag update, continue rendering after mode teardown → use-after-free crash.
  - **Fix Applied**: Changed `_otaMode` to `std::atomic<bool>` with proper loads/stores.

- **Push OTA failure path never restores display**
  - **Source**: Validator B
  - **File**: `firmware/adapters/WebPortal.cpp:692-709`
  - **Evidence**: After `prepareForOTA()` succeeds (line 656), later failures (magic byte, partition, begin) only abort Update and return error, but never call `ModeRegistry::completeOTAAttempt(false)`. Display task stays in OTA mode forever.
  - **Impact**: Device soft-stuck on "Updating..." after upload failure until power cycle.
  - **Fix Applied**: Added `ModeRegistry::completeOTAAttempt(false)` calls to all failure paths after `prepareForOTA()` succeeds.

- **OTAFailurePhase defined but never set**
  - **Source**: Validator A
  - **File**: `firmware/core/OTAUpdater.h:40-46`, `firmware/core/OTAUpdater.cpp`
  - **Evidence**: `_failurePhase` enum exists and has getter methods, but code inspection shows it IS populated in download task proc (lines 563, 594, 611, 643, 681, 700, 768, 792, 809). Validator A claimed it's never set — FALSE.
  - **Verification**: Re-read `OTAUpdater.cpp:543-838` confirms all error paths DO set `_failurePhase` appropriately (DOWNLOAD or VERIFY).
  - **Verdict**: **FALSE POSITIVE - no fix needed**, but added clarifying comment for dl-7.3 readiness.

## Suggested Future Improvements

- **Scope**: Add largest-contiguous-block check alongside heap threshold
  - **Rationale**: Current check only verifies total free heap, not fragmentation. On heavily fragmented heap, `malloc()` can fail even with sufficient total bytes.
  - **Effort**: Low — call `heap_caps_get_largest_free_block(MALLOC_CAP_8BIT)` alongside `ESP.getFreeHeap()`.

- **Scope**: Bundle Let's Encrypt root CA for GitHub HTTPS verification
  - **Rationale**: Current `setInsecure()` usage is acceptable MVP tradeoff but exposes MITM risk.
  - **Effort**: Medium — requires embedding CA certificate in firmware and verifying against api.github.com chain.

- **Scope**: Progress reporting for .sha256 file download
  - **Rationale**: Currently progress stays at 0% until binary download starts. Setting `_progress = 1` after SHA fetch would provide visual feedback.
  - **Effort**: Trivial — one line: `_progress = 1;` after SHA download completes.
