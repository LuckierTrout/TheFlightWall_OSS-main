## Issues Dismissed

- **Claimed Issue**: `schedule_active` in `/api/status` should be changed to `isScheduleDimming()` | **Raised by**: Validators A, C | **Dismissal Reason**: Story Task 4.2 and the fn-2.1 Review Follow-ups section both explicitly define `schedule_active` as "enabled + NTP synced = scheduler can run" and `schedule_dimming` as the real-time override state. Changing `schedule_active` would be a breaking API change outside story scope. The new `schedule_dimming` field resolves the semantic gap as designed.

- **Claimed Issue**: `ScheduleConfig` copies `String timezone` in `tickNightScheduler()` every second — ~2.6M heap allocations over 30 days | **Raised by**: Validator A | **Dismissal Reason**: The fix (a new `getScheduleDimmingParams()` function) contradicts the story's "No new files / no new functions beyond what's specified" constraint. The 1/sec String copy is a known ESP32 heap-fragmentation risk but is a future-story optimization concern, not a blocking bug for this story's NFR.

- **Claimed Issue**: `g_lastNtpSynced` (plain bool) race with `g_ntpSynced` (atomic) can produce duplicate/missing transition log lines | **Raised by**: Validator A | **Dismissal Reason**: `g_lastNtpSynced` is read and written exclusively by `tickNightScheduler()` on Core 1's Arduino loop task. The SNTP callback also runs on Core 1 but on the LWIP task — preemption is possible but the consequence is at most a duplicate or missing log line. The story's own comment documents "Core 1 only — no atomic needed". Accepted design risk for a non-functional log side-effect.

- **Claimed Issue**: Redundant `ConfigManager::getSchedule()` at lines 327 and 363 | **Raised by**: Validator B | **Dismissal Reason**: FALSE POSITIVE. Reading the actual code at lines 326–365, line 327 is `DisplayConfig newDisp = ConfigManager::getDisplay()` — not a `getSchedule()` call. There is exactly one `getSchedule()` call at line 363, inside the `if (g_schedDimming.load())` guard, so it only executes when actually dimming.

- **Claimed Issue**: Race between config read (line 356) and `g_schedDimming.load()` check (line 362) — one frame of wrong brightness if schedule transitions mid-handler | **Raised by**: Validator B | **Dismissal Reason**: The race window is microseconds; consequence is at most one display frame (~50ms) of wrong brightness, self-correcting on the next scheduler tick (1s) or config change. This is an accepted design risk for a transient display artifact — no functional correctness or data integrity issue.

- **Claimed Issue**: `g_ntpSynced` never resets to `false` after WiFi disconnect — scheduler silently continues post-disconnect | **Raised by**: Validator C | **Dismissal Reason**: Already adjudicated FALSE POSITIVE in fn-2.1 antipatterns (Rounds 1, 2, and 3). AC #4 defines `ntpSynced=false` only as the pre-first-sync default. "Sticky true" post-sync is the intentional contract. WiFi-reconnect NTP invalidation semantics are a future-story concern.

- **Claimed Issue**: AC #6 behavioral mismatch — manual brightness is re-overridden immediately (in the same config-change handler), not "on the next scheduler tick" as the story describes | **Raised by**: Validator C | **Dismissal Reason**: Intentional documented deviation. The story Dev Notes ("Brightness Override Mechanism") explicitly say: "because they trigger `g_configChanged`, the display task's handler runs immediately and re-applies the dim override (not 'on the next scheduler tick' — it's immediate, which is better than AC#6 describes)." The code comment at line 360 also documents this. Not a bug.

- **Claimed Issue**: Unit tests only test a local `isInDimWindow()` helper, never calling `tickNightScheduler()` or asserting `g_schedDimming` transitions | **Raised by**: Validators B, C | **Dismissal Reason**: Architectural limitation already established across fn-2.1 reviews. Unity tests on ESP32 cannot instantiate the main loop, LWIP scheduler, or FastLED display. The local `isInDimWindow()` helper is the production arithmetic extracted as a testable unit — this is correct TDD discipline. The story explicitly scopes "unit tests validate the exact same arithmetic the scheduler uses." Full integration testing requires hardware.

- **Claimed Issue**: Story file list drift — `WebPortal.cpp` listed but untouched; `ConfigManager.cpp` unlisted but changed | **Raised by**: Validator C | **Dismissal Reason**: Audit trail concern, not a product defect. Fixes applied in this synthesis bring `WebPortal.cpp` in line with the story. `ConfigManager.cpp` change (NVS validation) is a synthesis-added defensive fix beyond the original story spec.
