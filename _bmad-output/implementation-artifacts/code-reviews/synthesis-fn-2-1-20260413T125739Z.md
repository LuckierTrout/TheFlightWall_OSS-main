### High

- **Issue**: Hot-reload TZ guard fires on first unrelated config write after boot — `s_lastAppliedTz` was declared `static String s_lastAppliedTz;` **inside** the lambda, initialising to `""`. On first `onChange` trigger (e.g., brightness write), `sched.timezone` = `"UTC0"` ≠ `""` → `configTzTime()` restarts SNTP, discarding any in-flight NTP sync. The antipatterns doc said this was "fixed," but the fix was only partial.
  | **Source**: Validators B, C | **File**: `firmware/src/main.cpp:659` | **Fix**: Moved `s_lastAppliedTz` outside the lambda as a `static` seeded with `ConfigManager::getSchedule().timezone` before `onChange` registration; lambda now captures by reference `[&s_lastAppliedTz]`.

- **Issue**: Empty timezone string accepted and forwarded to `configTzTime("")` — ConfigManager's `updateConfigValue()` for `"timezone"` checked only type and max-length (>40), not min-length (>0). A `POST /api/settings` with `{"timezone":""}` would be stored and passed to `configTzTime("")`, which calls `tzset("")` — silently falling back to UTC with no feedback path.
  | **Source**: Validators B, C | **File**: `firmware/core/ConfigManager.cpp:138` | **Fix**: Added `if (tz.length() == 0) return false;` before the max-length check. Added regression test case in `test_apply_json_schedule_validation`.
