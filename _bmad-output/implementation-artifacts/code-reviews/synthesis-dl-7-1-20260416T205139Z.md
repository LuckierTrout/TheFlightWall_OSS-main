## Issues Verified (by severity)

### Critical

1. **Pull OTA ignores `prepareForOTA()` failure and flashes anyway**  
   **Source**: Validator A  
   **File**: `firmware/core/OTAUpdater.cpp:457`  
   **Evidence**: Code logs "prepareForOTA returned false — continuing anyway" then proceeds to `Update.begin()` and flash writes. AC2 required OTA to abort if quiesce fails.  
   **Fix**: Changed to abort download task with ERROR state when `prepareForOTA()` returns false.

2. **Failed push OTA never exits OTA mode**  
   **Source**: Validator A  
   **File**: `firmware/adapters/WebPortal.cpp:656-714`  
   **Evidence**: Upload handler calls `prepareForOTA()` but no failure paths call `completeOTAAttempt(false)`, leaving device stuck showing "Updating..." until reboot.  
   **Fix**: Added `ModeRegistry::completeOTAAttempt(false)` calls to all upload error paths after `prepareForOTA()` succeeds.

3. **`prepareForOTA()` violates Core 0 ownership boundary (use-after-free risk)**  
   **Source**: Validator A  
   **File**: `firmware/core/ModeRegistry.cpp:472-501`  
   **Evidence**: `prepareForOTA()` deletes `_activeMode` from Core 1 while Core 0 display task may still be inside `tick()` or `render()`. Story AC2 required coordination with Core 0.  
   **Fix**: Added atomic handshake. Core 1 sets `_otaPrepareRequested` flag, Core 0 checks in `displayTask()` before mode rendering, performs teardown, sets `_otaPrepareCompleted`. Core 1 spins waiting for completion (with timeout).

### High

4. **`Update.end(true)` failure path does not call `Update.abort()`**  
   **Source**: Validator A  
   **File**: `firmware/core/OTAUpdater.cpp:806-817`  
   **Evidence**: After `Update.begin()` succeeds, all error paths must call `Update.abort()` per AC8. The `Update.end(true)` failure branch exits task without aborting.  
   **Fix**: Added `Update.abort()` call before setting ERROR state in `Update.end()` failure path.

5. **`esp_ota_set_boot_partition()` return value ignored**  
   **Source**: Validator A  
   **File**: `firmware/core/OTAUpdater.cpp:820`  
   **Evidence**: AC7 required verification that boot slot was actually selected before reboot. Code ignores return value.  
   **Fix**: Added `esp_err_t` check. If `esp_ota_set_boot_partition()` fails, abort with ERROR state and call `completeOTAAttempt(false)`.

6. **Race condition on `_downloadTaskHandle` (missing synchronization)**  
   **Source**: Validator B  
   **File**: `firmware/core/OTAUpdater.cpp:489`  
   **Evidence**: `startDownload()` checks `_downloadTaskHandle != nullptr` without mutex/atomic protection. Task clears handle before `vTaskDelete()`. Rapid concurrent calls could observe stale handle.  
   **Fix**: Changed `_downloadTaskHandle` to `std::atomic<TaskHandle_t>` and use `.load()` / `.store()` for all access.

## Issues Dismissed

**TLS validation disabled (setInsecure)**  
**Claimed by**: Both Validator A and B as "Critical Security Vulnerability"  
**Files**: `firmware/core/OTAUpdater.cpp:498, 579, 665`  
**Dismissal Reason**:  
This is **documented technical debt** acknowledged in code comments and story completion notes. ESP32 Arduino does not bundle GitHub API or GitHub CDN root CAs by default. Adding full CA validation requires embedding Let's Encrypt root cert (~2KB) and maintaining cert expiry updates. Story dl-7.1 AC#2 explicitly documented this tradeoff: "setInsecure() acceptable per AC #2 tradeoff."

SHA-256 integrity verification still protects against **accidental corruption** and **compromised release artifacts** (attacker must control both firmware binary and `.sha256` file on GitHub CDN simultaneously). For a hobbyist ESP32 project pulling from public GitHub releases, this is an acceptable risk vs. complexity tradeoff.

**Future improvement**: Add CA bundle validation in a dedicated security hardening story, not as a regression against dl-7.1 acceptance criteria.
