## Issues Dismissed

- **Claimed Issue**: `fillScreen(0)` violates AC5 zone bounds | **Raised by**: Reviewer A | **Dismissal Reason**: ClassicCardMode uses identical `ctx.matrix->fillScreen(0)` at the same position (line 95) and was reviewed/accepted in ds-1.4. The architecture uses a single active mode on Core 0; `fillScreen` is a canvas reset, not "drawing content outside zones." AC5 governs content drawing, not clearing. The reviewer's suggested alternative (clearing individual zones + adding fillRect to fallback/loading paths) contradicts the accepted reference pattern.

- **Claimed Issue**: `clampZone` fails mathematically for negative x/y coordinates | **Raised by**: Reviewers A & B | **Dismissal Reason**: `Rect` fields (`x`, `y`, `w`, `h`) are all `uint16_t` (confirmed in `firmware/core/LayoutEngine.h`). Negative coordinates are physically impossible in a `uint16_t` field. The reviewer's "if `c.x` is `-10`" scenario cannot occur. ClassicCardMode has the same `clampZone` implementation.

- **Claimed Issue**: Zero real-matrix rendering tests | **Raised by**: Reviewer B | **Dismissal Reason**: Valid test quality gap, but on-device ESP32 test infrastructure requires physical hardware + real FastLED_NeoMatrix. Cannot be resolved in code-review synthesis without hardware access. Deferred as future improvement.

- **Claimed Issue**: No zone clamping / bounds tests | **Raised by**: Reviewer B | **Dismissal Reason**: Same constraint as above — physical ESP32 hardware required. Deferred.

- **Claimed Issue**: `renderFlightZone` omits aircraft info (content regression) | **Raised by**: Reviewer B | **Dismissal Reason**: ds-2.1 AC#1 explicitly specifies flight zone content as "airline name and route" only — aircraft type is not listed. The story deliberately focuses on telemetry enrichment (altitude, speed, heading, vertical rate). This is not a regression from the story spec; it's a difference from ClassicCardMode that is intentional per the AC.

- **Claimed Issue**: `telLine2` drawn unconditionally even when potentially empty | **Raised by**: Reviewer B | **Dismissal Reason**: `formatTelemetryValue` always produces `"--"` for NaN values (per existing behavior) and numeric strings for valid values — `telLine2` cannot be empty in practice. ClassicCardMode has identical logic with an explicit comment: "telLine2 is always set when linesAvailable >= 2" (line 266).

- **Claimed Issue**: `renderLoadingScreen` computes negative Y for very small matrices | **Raised by**: Reviewer B | **Dismissal Reason**: Pre-existing pattern identical in ClassicCardMode (line 347). Not a regression introduced by ds-2.1. Should be fixed in both modes together if addressed.

- **Claimed Issue**: 1-line compact path has no aircraft fallback | **Raised by**: Reviewer B | **Dismissal Reason**: The precedence chain always resolves to `f.operator_code.c_str()` as final fallback — this should never be empty for a valid `FlightInfo`. Furthermore, ds-2.1 AC#1 does not list aircraft in the flight zone.

- **Claimed Issue**: Test helpers don't exercise airline precedence chain | **Raised by**: Reviewer B | **Dismissal Reason**: Valid test quality gap. Deferred as a future improvement for the ds-2.2 test expansion cycle.

- **Claimed Issue**: Dead `#include <cmath>` | **Raised by**: Reviewer A | **Dismissal Reason**: Identical include exists in ClassicCardMode (line 20). Pre-existing accepted pattern. May provide transitive include guarantees for ESP32 headers.
