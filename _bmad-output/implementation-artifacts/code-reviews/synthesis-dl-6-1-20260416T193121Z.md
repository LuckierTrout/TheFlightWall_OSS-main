## Issues Verified (by severity)

### Critical

**1. [SECURITY VULNERABILITY] Unauthenticated GitHub release discovery enables MITM supply-chain attacks**
- **Source:** Reviewer A (primary), Reviewer B (performance concern only)
- **Evidence:** `firmware/core/OTAUpdater.cpp:345`, `:579`, `:665` - All GitHub HTTPS requests use `client.setInsecure()`
- **Impact:** Any on-path attacker can forge `tag_name`, release notes, and asset URLs. With dl-7 download code already present in the implementation (scope violation), this becomes an immediate firmware supply-chain vulnerability, not just a cosmetic version-check issue.
- **Fix:** This is a **DOCUMENTED TRADEOFF** in AC #2 and story completion notes ("setInsecure() acceptable per AC #2 tradeoff... Future: bundle Let's Encrypt root CA for production hardening"). The story explicitly defers CA root bundling as out-of-scope for dl-6.1. This is a **VALID FINDING** but is **ACCEPTED TECHNICAL DEBT** per the story design. The reviewers correctly identified the risk; the story intentionally defers hardening to future work.
- **Verdict:** **DEFER** - Acknowledged in story documentation; requires dedicated security hardening story.

**2. [RACE CONDITION] `_downloadTaskHandle` torn reads/writes across Core 1 (loop/HTTP) and download task**
- **Source:** Reviewer B (primary), Reviewer A (broader shared-state issue)
- **Evidence:** `firmware/core/OTAUpdater.cpp:637` - `_downloadTaskHandle` set to `nullptr` inside task after completion; `startDownload()` checks handle on Core 1 without synchronization
- **Impact:** If `startDownload()` is called between task completion and handle clearing, stale handle could be dereferenced or double-start logic could fail. Also affects `_state`, `_progress`, `_failurePhase`, `_lastError`, all asset URL buffers (lines 31-42) exposed via raw getters (77-96) while WebPortal async handlers read them (WebPortal.cpp:1452-1463) and download task mutates them (766-824).
- **Reproduction:** Call `startDownload()` repeatedly during task exit window; or access `/api/ota/status` during active download.
- **Fix:** Add mutex/critical section around shared OTA state, or use atomic state transitions with memory barriers.
- **Verdict:** **CRITICAL DEFECT** - dl-6.1 only exposes read-only accessors, but dl-7 download task (already present in code) writes state concurrently. This is a **REAL BUG** requiring immediate fix.

### High

**3. [INCOMPLETE API CONTRACT] GitHub rate-limit handling only detects HTTP 429, ignores 403**
- **Source:** Reviewer A
- **Evidence:** `firmware/core/OTAUpdater.cpp:363-369` special-cases only 429; `:381-387` collapses 403 into generic HTTP failure
- **Impact:** GitHub REST API documents primary and secondary rate limits can surface as `403 X-RateLimit-Remaining: 0` or `429`. Users lose "Rate limited — try again later" messaging when 403 rate-limit occurs.
- **Reference:** [GitHub REST API Rate Limits](https://docs.github.com/en/rest/using-the-rest-api/rate-limits-for-the-rest-api?apiVersion=latest)
- **Fix:** Check both `httpCode == 429` and presence of `X-RateLimit-Remaining: 0` header for 403 responses.
- **Verdict:** **VALID** - AC #6 says "HTTP 429" but GitHub API contract requires checking both.

**4. [INCOMPLETE FALLBACK] Asset `url` field parsing incomplete for GitHub asset API URLs**
- **Source:** Reviewer A
- **Evidence:** `firmware/core/OTAUpdater.cpp:215`, `:290` store asset API `url` field; downloader at `:667-674` performs plain GET with only `User-Agent`
- **Impact:** GitHub asset API `url` field requires `Accept: application/octet-stream` header or redirects to `browser_download_url`. Current implementation would fail when `browser_download_url` is absent and only `url` is available.
- **Reference:** [GitHub Releases Assets API](https://docs.github.com/en/rest/releases/assets?apiVersion=2026-03-10)
- **Fix:** Add `Accept: application/octet-stream` header to binary download request, or remove `url` fallback if not intended to support asset API endpoints.
- **Verdict:** **VALID** - AC #4 explicitly requires "url fallback" but current implementation is incomplete for GitHub's asset API contract.
