## 🏛️ Architectural Sins

- **[7/10] Dependency Inversion Violation:** `WebPortal.cpp` declares `extern bool isNtpSynced();` — a direct dependency on a global function defined in `main.cpp`. In the hexagonal architecture, adapters should depend on interfaces or core APIs, not on globals in the bootstrap module. This makes unit-testing the web adapter in isolation impossible without linking main.cpp.
  - 📍 `firmware/adapters/WebPortal.cpp:41`
  - 💡 Fix: Add `isNtpSynced()` as a static method on `SystemStatus` or expose it via a small `NtpTimeProvider` interface that WebPortal receives at construction.

- **[6/10] Open/Closed Violation:** `TZ_MAP` is copy-pasted into both `dashboard.js` and `wizard.js`. Adding a new timezone or correcting a DST rule requires editing two files and regenerating two `.gz` bundles. The map should live in `common.js` (or a shared module) and be imported by both consumers.
  - 📍 `firmware/data-src/dashboard.js` (top), `firmware/data-src/wizard.js` (top)
  - 💡 Fix: Move `TZ_MAP` and `getTimezoneConfig()` to `common.js`; include `common.js` in the wizard HTML load order.

- **[5/10] Leaky Abstraction:** `schedule_active` in the `/api/status` JSON response is computed as `sched_enabled && ntp_synced`. The field name implies real-time schedule execution state (i.e., "the device is currently inside the dim window"), but it actually returns feature-enablement plus time-sync readiness. API consumers will misinterpret this.
  - 📍 `firmware/adapters/WebPortal.cpp:787`
  - 💡 Fix: Rename to `schedule_enabled` until fn-2.2 implements actual in-window calculation, or compute the real state using the existing `timeInWindow()` logic.

## 🐛 Correctness & Safety

- **🐛 Bug: Stale `g_ntpSynced` flag after SNTP restart**
  - 📍 `firmware/src/main.cpp` (callback), `firmware/adapters/WiFiManager.cpp:132-134`, `firmware/src/main.cpp:~624`
  - **Reproduction:**
    1. Boot device, connect to WiFi, wait for NTP sync → `g_ntpSynced = true`, status = "Clock synced".
    2. Disconnect WiFi (device stays powered).
    3. Reconnect to a network where NTP is firewalled/unreachable.
    4. `WiFiManager::_onConnected()` calls `configTzTime()`, which restarts SNTP.
    5. SNTP fails to reach servers.
    6. `g_ntpSynced` **remains `true`** and status stays "Clock synced" indefinitely.
  - **Impact:** Directly contradicts AC #3 ("When sync attempt fails, ntpSynced remains false") for any reconnect scenario. Dashboard and scheduler believe time is authoritative when it may be stale RTC drift.

- **🐛 Bug: Silent UTC fallback for invalid/missing timezone mappings**
  - 📍 `firmware/data-src/dashboard.js:101`, `firmware/data-src/wizard.js` (identical `getTimezoneConfig()`)
  - **Reproduction:** Browser is set to `Asia/Manila` or `Europe/Zagreb` — zones not in `TZ_MAP`. `getTimezoneConfig()` returns `{ iana: "Asia/Manila", posix: "UTC0" }`. When the wizard/dashboard eventually sends this to the ESP32, the device runs on UTC with zero user-facing warning.
  - **Impact:** Users in unmapped regions get incorrect local time. The function gives callers no way to detect the fallback.

- **🔒 [MEDIUM] Security / Robustness: No POSIX timezone validation in ConfigManager**
  - 📍 `firmware/core/ConfigManager.cpp` (applyJson validation for `timezone` key)
  - **Impact:** `applyJson` accepts any string ≤40 chars. A user could POST `"timezone": "NOT_A_VALID_TZ"` and `configTzTime()` would silently fall back to UTC via `tzset()`. There is no feedback loop to the user that their timezone string is malformed. While not a direct security vulnerability, it is a trust-boundary violation where user input is passed unvalidated to a C library function that affects system clock behavior.
