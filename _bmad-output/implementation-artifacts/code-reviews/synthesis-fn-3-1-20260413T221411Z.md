## Issues Dismissed

- **Claimed Issue**: Race condition in `enterStep6()` — "async callback uses stale `currentStep`; should capture step into a variable" | **Raised by**: Reviewer A | **Dismissal Reason**: The current `if (currentStep === 6)` check is *correct*. If the user clicks "No — take me back" before the `/api/positioning/start` callback fires, `currentStep` becomes 4, and the guard correctly suppresses the Step 5 redirect. Reviewer A's suggested fix (capturing step into `enterStep`) would check `enterStep === 6` (always true at callback time), causing the back-navigation to incorrectly overwrite the user's already-completed "No" action with a forced Step 5 jump.

- **Claimed Issue**: Red corner marker should render on tile 0 only; current canvas renders it on every tile | **Raised by**: Reviewer B | **Dismissal Reason**: Directly contradicted by firmware source. `NeoMatrixDisplay::renderPositioningPattern()` lines 747-752 are **inside** the nested `for (row) / for (col)` loop — the red corner marker is drawn on *every* tile. The antipatterns document that claimed "tile 0 only" was itself incorrect. The canvas implementation is correct.

- **Claimed Issue**: Canvas preview ignores `origin_corner`, `scan_dir`, `zigzag` — browser disagrees with wall for non-default wiring | **Raised by**: Reviewers B, D | **Dismissal Reason**: Both firmware and canvas use `gridIdx = row * tilesX + col` (visual row-major order). The NeoMatrix library handles the physical LED strand remapping internally per wiring flags. Tile *numbers* are always assigned in visual order; only the physical pixel addressing changes. The canvas correctly matches the firmware's visual numbering.

- **Claimed Issue**: `saveAndConnect()` SOLID violation — does too many things; should split into `buildSavePayload()`, `handleReboot()`, etc. | **Raised by**: Reviewer A | **Dismissal Reason**: Pre-existing function pattern that predates fn-3.1. Architectural refactoring is out of scope for a code review targeting this story's delta.

- **Claimed Issue**: CORS wildcard `Access-Control-Allow-Origin: '*'` in mock server is a security vulnerability | **Raised by**: Reviewer A | **Dismissal Reason**: The mock server is a **local test-only tool** (`tests/e2e/mock-server/server.ts`), never shipped on the device. `*` is standard for local dev servers. Not a defect in product code.

- **Claimed Issue**: No CSRF protection on `/api/settings` | **Raised by**: Reviewer A | **Dismissal Reason**: The device operates exclusively in AP mode (isolated WiFi access point, no internet routing). Cross-origin requests are physically impossible in this network topology. Not applicable.

- **Claimed Issue**: RGB test produces tinted text instead of solid-color fill — AC4 requires "solid red → solid green → solid blue" | **Raised by**: Reviewer B | **Dismissal Reason**: Explicitly dismissed in the antipatterns document: "by design per Dev Notes; sufficient to verify each RGB channel independently; no firmware changes required per AC constraints." This dismissal pre-dates this synthesis pass.

- **Claimed Issue**: AC5 completion text "Your FlightWall is ready!" is never shown | **Raised by**: Reviewer D | **Dismissal Reason**: Partially incorrect. `showHandoff()` replaces the entire wizard DOM with content including `<p>Your FlightWall is ready! Fetching your first flights...</p>`. The AC5 copy IS shown — it is the handoff screen. The "Rebooting…" gap was the real defect (fixed above).

- **Claimed Issue**: `rebootRequested` flag makes post-reboot network loss appear as success | **Raised by**: Reviewer D | **Dismissal Reason**: This is intentional and documented in the catch handler: "Network loss after requesting reboot is expected because the device is leaving AP mode." The device drops the TCP connection when it leaves AP mode; treating that disconnect as successful handoff is correct behavior.

- **Claimed Issue**: No E2E test specs for Step 6 | **Raised by**: Reviewers B, D | **Dismissal Reason**: Explicitly deferred per Dev Agent Record: "Full E2E test cases for Step 6 not written in this pass — page-object infrastructure updated; dedicated test spec creation belongs in a separate test story."

- **Claimed Issue**: `validateStep(4)` doesn't validate `origin_corner`, `scan_dir`, `zigzag` | **Raised by**: Reviewers B, C | **Dismissal Reason**: These are `<select>` elements whose values are browser-constrained to the provided `<option>` values (0 or 1). No user input can produce an out-of-range value without DOM manipulation, and firmware validates all NVS writes. Low-risk gap that doesn't warrant validation code.
