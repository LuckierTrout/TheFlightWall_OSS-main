## 🐛 Correctness & Safety

- **🐛 Bug:** Bricked UI on early-exit success path network failure
  - 📍 `firmware/data-src/dashboard.js:940`
  - 🔄 Reproduction: If the POST response indicates immediate success (`switch_state === 'idle'`), `finalizeModeSwitch` is called with `skipReload` false, bypassing `clearSwitchingStates()`. It immediately calls `loadDisplayModes()`. If this GET request fails due to a network error, the Promise catches and returns `null`, and the UI cards are permanently left with `.switching` and `.disabled` (`pointer-events: none`) classes. The user is locked out of interacting with the cards until a full page reload.

- **🐛 Bug:** Race condition unlocking concurrent switches prematurely
  - 📍 `firmware/data-src/dashboard.js:925`
  - 🔄 Reproduction: In the error paths of `switchMode` (HTTP error, timeout, network failure), `modeSwitchInFlight` is set to `false` synchronously before the async `loadDisplayModes()` call finishes. This allows the user to initiate a new mode switch while the old one is still recovering. When the delayed `loadDisplayModes` resolves, its `renderModeCards()` call rebuilds the DOM, wiping out the `.switching` and `.disabled` visual states of the newly initiated switch.

- **🐛 Bug:** Race condition in scoped error auto-dismiss
  - 📍 `firmware/data-src/dashboard.js:845`
  - 🔄 Reproduction: `showCardError` creates an auto-dismiss timeout that removes the error alert after 5 seconds. However, `clearCardError(card)` removes *any* existing alert in the DOM without clearing the previous timer. If `showCardError` is called twice in quick succession on the same card, the first timeout will fire and prematurely clear the second error message before its 5 seconds are up.

- **🐛 Bug:** Premature removal of active state during switch (AC 3 Violation)
  - 📍 `firmware/data-src/dashboard.js:895`
  - 🔄 Reproduction: In `switchMode()`, `.active` is removed from ALL cards (including the previously active one) immediately upon click. This directly violates AC 3 which states the previously active card loses active chrome "Given the firmware confirms success, When the client determines the switch completed". This creates a degraded intermediate visual state where no card appears active during the 1-2 second switch.

- **🐛 Bug:** Reduced motion opacity violation (AC 5)
  - 📍 `firmware/data-src/style.css:350`
  - 🔄 Reproduction: AC 5 explicitly mandates "use static var(--primary) ... at full opacity" for reduced motion users. The `.mode-card.switching` base class sets `opacity: 0.7` on the entire card. The `@media (prefers-reduced-motion: reduce)` block overrides the animation and border colors but fails to set `opacity: 1`, leaving the card dimmed and violating the accessibility requirement.

## 🛠️ Suggested Fixes

### 1. Fix Bricked UI on early-exit failure

**File:** `firmware/data-src/dashboard.js`
**Issue:** Clear switching states before attempting the reload to guarantee safe UI state if the network drops.

**Corrected code:**
```javascript
    if (skipReload) {
      doFocus();
    } else {
      clearSwitchingStates(); // Prevent UI lock-in if loadDisplayModes fails
      // Reload full card state to get correct active chrome (early-exit path)
      loadDisplayModes().then(doFocus);
    }
```

### 2. Fix Race condition unlocking switches prematurely

**File:** `firmware/data-src/dashboard.js`
**Issue:** Keep `modeSwitchInFlight` locked until the DOM is successfully rebuilt.

**Corrected code:**
```javascript
      // HTTP error from firmware (AC #8)
      if (!res.body || !res.body.ok) {
        clearSwitchingStates();
        var errMsg = (res.body && res.body.error) ? res.body.error : 'Mode switch failed';
        // Reload first (rebuilds DOM), then attach error to the freshly rendered card
        loadDisplayModes().then(function() {
          modeSwitchInFlight = false; // Unlock ONLY after recovery
          showCardError(getModeCard(modeId), errMsg);
        });
        return;
      }
```
*(Note: Apply this identical fix to the timeout and `.catch()` network failure blocks as well).*
