## 🏛️ Architectural Sins

- **[10/10] Open/Closed Principle (SOLID):** `ConfigManager::setModeSetting` hardcodes validation for specific modes (`clock` and `depbd`). This defeats the purpose of the dynamic `settingsSchema` architecture and forces `ConfigManager.cpp` to be modified for every new mode added. *(Note: The story Task 1.2 explicitly requested this, but it is a severe architectural anti-pattern).*
  - 📍 `firmware/core/ConfigManager.cpp:469`
  - 💡 Fix: Remove mode-specific validation from `ConfigManager`. API handlers in `WebPortal.cpp` should dynamically validate incoming requests against the mode's `ModeSettingsSchema` (`minValue`/`maxValue`) BEFORE saving.

- **[10/10] Abstraction Boundary Breach (Rule 18):** `ClockMode::render` executes `ConfigManager::getModeSetting` on the rendering hot path. The architecture explicitly commands: *"18. Modes must ONLY access data through RenderContext — zero ConfigManager reads from inside a mode"* and *"Modes → ConfigManager | Via init() only"*.
  - 📍 `firmware/modes/ClockMode.cpp:76`
  - 💡 Fix: Remove the `getModeSetting` call from `render()`. Rely exclusively on the cached `_format` value read during `init()`. Accept that the format won't hot-reload instantly without a mode switch, as designed by the architecture.

---

## ⚡ Performance & Scalability

- **[High] NVS Flash I/O on Display Task:** Because `ClockMode::render()` re-reads the NVS setting every time the second changes, it triggers the `Preferences` library to access flash memory on Core 0. This can block the CPU for milliseconds and cause LED rendering jitter.
  - 📍 `firmware/modes/ClockMode.cpp:76`
  - 💡 Fix: Same as Rule 18 violation — move reads strictly to `init()`.

## 🐛 Correctness & Safety

- **🐛 Bug:** `ClockMode` dynamic text scaling causes the entire clock to suddenly shrink and jump positions twice a day. In 12-hour format, "9:59AM" is 6 characters (triggering scale 4), but "10:00AM" is 7 characters (forcing a shrink to scale 3). The UI will violently bounce at 10:00 and 1:00.
  - 📍 `firmware/modes/ClockMode.cpp:111`
  - 🔄 Reproduction: Set clock format to 12-hour. Observe the transition from 9:59 AM to 10:00 AM on the matrix.
  - 💡 Fix: For 12-hour format, pad the hour with a space (`%2d:%02d%s`) so the string length remains constant at 7 characters, preventing layout jumping.

- **🎭 Lying Test:** `test_set_mode_setting_validates_clock_format_range`
  - 📍 `firmware/test/test_config_manager/test_main.cpp:518`
  - 🤥 Why it lies: This test actively enforces the OCP architectural violation by demanding that `ConfigManager` knows about the internal format ranges of the "clock" mode. It locks in bad design and forces future maintainers to break the test to fix the architecture.

## 🔧 Maintainability Issues

- **💣 Tech Debt:** `CLOCK_SETTINGS` and `CLOCK_SCHEMA` are declared as `static const` inside the header file.
  - 📍 `firmware/modes/ClockMode.h:15`
  - 💥 Explosion radius: Every `.cpp` file that includes this header (`main.cpp`, `ClockMode.cpp`, tests) allocates its own discrete copy of these structs in flash memory. This causes firmware bloat and introduces subtle pointer-inequality bugs if schemas are ever compared by reference.
  - 💡 Fix: Declare them as `extern const ModeSettingsSchema CLOCK_SCHEMA;` in the header, and define the arrays in `ClockMode.cpp`.
