**Findings**

1. `Critical` Reboot recovery is falsely optimistic. [common.js](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/common.js:25) uses plain `fetch()` with default cache behavior, and [dashboard.js](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/dashboard.js:2830) treats the first `200 /api/status` as proof that the new firmware is back. That is not a valid success criterion. A cached response, a pre-reboot response, or any stale `200` can trigger “Updated to v…” before the new image is actually running. The client needs `cache: 'no-store'` and a real reboot proof such as uptime reset or version/boot-id change.

2. `Important` The 60-second polling loop is a resource leak in slow-failure cases. [dashboard.js](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/dashboard.js:2818) starts a new `FW.get('/api/status')` every 3 seconds, while [common.js](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/common.js:25) provides no request timeout or abort support. If the device is offline, mobile browsers can keep many fetches hanging concurrently. You still hit the UI timeout at 60s, but you do it with a pile of outstanding requests and undefined completion ordering.

3. `Important` Upload failure handling misclassifies a successful OTA as a hard error on weak links. The server intentionally reboots 500ms after responding on success in [WebPortal.cpp](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/adapters/WebPortal.cpp:496). The client-side [dashboard.js](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/dashboard.js:2782) treats any `xhr.onerror` as “Connection lost during upload” and resets the UI. If the firmware was already committed and the reboot closes the socket before the body is fully observed, the user gets a false failure instead of the reboot countdown/polling path.

4. `Important` AC #11 is not met. The new progress text in [style.css](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/style.css:562) renders white or near-white text over `#58a6ff`; the actual contrast is only `2.53:1` for `#fff` and `2.14:1` for `#e6edf3`, both below WCAG AA. The new buttons in [dashboard.html](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/dashboard.html:247) and [dashboard.html](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/dashboard.html:263) also have no explicit `min-width`/`min-height`, so the claimed `44x44` touch target is not guaranteed.

**Suggested Fixes**

1. Make reboot polling non-cacheable and deterministic: `cache: 'no-store'`, one in-flight request max, per-request timeout via `AbortController`, and success only after `device.uptime_ms` resets or another explicit reboot token changes.
2. Treat `xhr.onerror` after upload progress as an ambiguous completion state; pivot into reboot polling instead of hard-resetting the OTA UI.
3. Fix AC #11 properly: darken the progress text/fill combination to meet AA, and add explicit `min-width:44px; min-height:44px` to all new interactive elements.
4. Split unrelated `dl-1.5` work out of the `fn-1.6` audit trail, or update the story’s file list and “Files NOT to modify” section so the artifact matches reality.
5. Add automated coverage for `POST /api/ota/ack-rollback`, `GET /api/settings/export`, rollback-ack persistence, and the reboot polling state machine.
6. Block or warn on settings export when `applyBar` is visible, or export current form state client-side so the downloaded backup matches the UI.
