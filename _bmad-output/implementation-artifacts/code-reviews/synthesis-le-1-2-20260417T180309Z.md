## Issues Dismissed (false positives with reasoning)

**Reviewer A:**

- **Claimed Issue**: `switch (type)` without `default` causes UB for out-of-range enum values | **Raised by**: Reviewer A | **Dismissal Reason**: The switch has an explicit `case WidgetType::Unknown: return false;` and an explicit `return false;` after the closing brace. In C++11, a switch with no matching case executes no case body and falls through to the post-switch statement — this is defined behavior per the standard. No UB exists.

- **Claimed Issue**: Alignment tests only smoke-check return value, not pixel math | **Raised by**: Reviewer A | **Dismissal Reason**: The test file itself explicitly acknowledges this at line 212: "pixel-accurate assertions require a real framebuffer which we don't have in the test env." This is a documented limitation, not a test gap that can be fixed without hardware. The story scope (AC#4, Task 7) did not mandate pixel-accurate assertions.

- **Claimed Issue**: `dispatch(type, spec)` ignores potential mismatch between explicit `type` param and `spec.type` field | **Raised by**: Reviewer A | **Dismissal Reason**: The API is intentionally designed this way — the header comment at line 88-91 of `WidgetRegistry.h` documents: "type is passed explicitly (rather than reading `spec.type`) to keep the dispatcher honest about the contract and to allow future callers to force a type without mutating the spec." This is a deliberate architectural choice, not a bug.

- **Claimed Issue**: `delay(2000)` in test setup adds 2s per test run | **Raised by**: Reviewer A | **Dismissal Reason**: `delay(2000)` in `void setup()` is the standard project pattern — `test_mode_schedule/test_main.cpp` and `test_config_manager/test_main.cpp` both use `delay(2000)`. It is needed to allow USB-serial to stabilize before Unity output begins on hardware.

- **Claimed Issue**: `fillRect` coverage relies on grep, not a static_assert or compile-time check | **Raised by**: Reviewer A | **Dismissal Reason**: Grep gates are the established project QA pattern (the story's Task 8 completion notes reference `grep -r "fillRect" firmware/widgets/`). A `static_assert` cannot detect function calls by name; a compile-time check would require a macro wrapper around `fillRect` which would be more invasive than the current solution. The grep gate is sufficient for CI.

- **Claimed Issue**: `test_clock_cache_reuse` phase 2 "arm" step can fail silently if `getLocalTime` returns false | **Raised by**: Reviewer A | **Dismissal Reason**: The throttle behavior is timestamp-based, not success-based — `s_lastClockUpdateMs` advances on both success and failure (this was the fix from Round 1 synthesis). Phase 2 does prove the throttle: after the first dispatch arms the timestamp, the next 49 dispatches are within the same `millis()` window and will NOT enter the refresh branch regardless of NTP state. The assertion `getTimeCallCount() <= 1` holds.

- **Claimed Issue**: Heavy coupling — `WidgetRegistry.h` includes `DisplayMode.h` (FastLED, vector, flight layout) for every TU | **Raised by**: Reviewer A | **Dismissal Reason**: This is a valid future refactoring concern but not a defect for LE-1.2 scope. The story explicitly reuses the existing `RenderContext` from `DisplayMode.h` rather than creating a new context struct (Dev Notes: "RenderContext already exists in firmware/interfaces/DisplayMode.h. LE-1.2 does NOT create a new context struct"). Decoupling would require creating a slimmer context type, which is LE-1.3+ scope.

**Reviewer B:**

- **Claimed Issue**: ClockWidget cache throttle bug — `s_lastClockUpdateMs` only advances on success | **Raised by**: Reviewer B | **Dismissal Reason**: This bug was identified and fixed in the **previous synthesis round (v1.2)**. The current code (line 62 of `ClockWidget.cpp`) explicitly places `s_lastClockUpdateMs = nowMs` outside the `if (getLocalTime...)` block and includes a multi-line comment explaining exactly why. The reviewer was reviewing stale/pre-fix code.

- **Claimed Issue**: Task 8 uses non-existent `--without-testing` PlatformIO flag | **Raised by**: Reviewer B | **Dismissal Reason**: Verified empirically — `pio test --help` confirms `--without-testing` is a valid flag. Running the exact Task 8 command (`pio test -e esp32dev --filter test_widget_registry --without-uploading --without-testing`) completed successfully in this synthesis session.

- **Claimed Issue**: LogoWidget buffer OOB read if spec.w or spec.h > 16 | **Raised by**: Reviewer B | **Dismissal Reason**: The `drawBitmapRGB565` call passes `kLogoW=16, kLogoH=16` as the **bitmap dimensions** and `spec.w, spec.h` as the **zone dimensions**. Inside the function, `drawW` and `drawH` are clamped to the bitmap size (`if (zoneW < w) drawW = zoneW` / otherwise `drawW = w = 16`). The index computation `bitmap[(row + offsetY) * w + (col + offsetX)]` is always bounded by `w=16` and `h=16`. Maximum index = `15*16 + 15 = 255`, which is within `s_buf[256]`. No OOB read is possible.

- **Claimed Issue**: TextWidget `truncateToColumns` could overflow 48-byte output buffer when adding "..." ellipsis | **Raised by**: Reviewer B | **Dismissal Reason**: The actual `truncateToColumns` implementation (DisplayUtils.cpp lines 65-76) explicitly computes `maxPrefix = outLen - 4` and clamps `prefixLen` to `maxPrefix`. With `outLen=48`: max prefix = 44 chars, dots = 3, null = 1, total = 48 exactly. The function is internally bounds-safe regardless of input size. No overflow is possible.

- **Claimed Issue**: AC#8 cache test unverifiable without seeing actual test file | **Raised by**: Reviewer B | **Dismissal Reason**: The actual test file (`firmware/test/test_widget_registry/test_main.cpp`) was read in full. The two-phase `test_clock_cache_reuse` test (lines 145–175) is structurally sound. Phase 1 resets state and runs 50 dispatches — the throttle fires on dispatch #1 and cache hits on #2–50. Phase 2 arms the cache then runs 49 more dispatches within the same millis window. Both phases assert `getTimeCallCount() <= 1`.

- **Claimed Issue**: `drawBitmapRGB565` 0x0000 transparency behavior unverified — comment may be inaccurate | **Raised by**: Reviewer B | **Dismissal Reason**: The actual `DisplayUtils.cpp` implementation (line 134) contains `if (pixel != 0) { matrix->drawPixel(...); }`. The black-as-transparent behavior is confirmed in the source. The LogoWidget comment is accurate.
