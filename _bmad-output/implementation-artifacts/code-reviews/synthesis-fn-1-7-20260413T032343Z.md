## Issues Dismissed

- **Claimed Issue**: Race condition — `importFileInput.value = ''` cleared before FileReader processes file | **Raised by**: Reviewer A | **Dismissal Reason**: The `File` object is captured as a parameter (`handleImportFile(importFileInput.files[0])`) before `importFileInput.value = ''` runs. The value clear exists specifically to allow re-selection of the same file on next attempt. This is the standard browser pattern. No race exists.

- **Claimed Issue**: `display_mode` missing from `KNOWN_EXTRA_KEYS` | **Raised by**: Reviewer C | **Dismissal Reason**: `display_mode` does not exist in `ConfigManager::dumpSettingsJson()` nor `applyJson()`. Verified by reading all 29 keys emitted by `dumpSettingsJson()` (lines 482–520). Adding a non-existent key to `KNOWN_EXTRA_KEYS` would silently send it in the POST payload, causing a 400 error when ConfigManager rejects it as an unknown key.

- **Claimed Issue**: `KNOWN_EXTRA_KEYS` hardcoded list is a forward-compatibility/maintenance risk (CRITICAL/IMPORTANT) | **Raised by**: Reviewers A & C | **Dismissal Reason**: Already dismissed in previous synthesis rounds. The story explicitly prohibits new server-side endpoints. The `KNOWN_EXTRA_KEYS` list matches exactly the 17 extra keys in `dumpSettingsJson()`. Future keys are a known tradeoff documented in story Risk Mitigation section. Sending unknown keys to ConfigManager causes 400 errors, so speculative future keys must never be in this list.

- **Claimed Issue**: SRP violation — `wizard.js` handles too many responsibilities | **Raised by**: Reviewers A & C | **Dismissal Reason**: Pre-existing architecture, not introduced by fn-1.7. ESP32 embedded web UI uses plain IIFE-scoped JS without module support. Refactoring would require ES module infrastructure that doesn't exist on the device's LittleFS server.

- **Claimed Issue**: DRY violation — `state` object and `WIZARD_KEYS` array define the same 12 keys | **Raised by**: Reviewer A | **Dismissal Reason**: Pre-existing pattern. `state` is a value store; `WIZARD_KEYS` is an iteration list for import. They serve different purposes. The duplication is intentional and minimal.

- **Claimed Issue**: `TOTAL_STEPS` is a magic number | **Raised by**: Reviewer A | **Dismissal Reason**: Pre-existing, not introduced by fn-1.7. Out of scope.

- **Claimed Issue**: Missing automated E2E test coverage for import scenarios | **Raised by**: Reviewer A | **Dismissal Reason**: Valid tech debt but out of scope for this synthesis. The WizardPage.ts page object has been substantially corrected in this round, providing the infrastructure for future wizard specs. Creating a wizard spec requires mock-server support for the AP-mode endpoints — a separate effort.

- **Claimed Issue**: Extra-key type validation allows malformed values through (e.g., `timezone: 123`) causing server-side 400 rejection after a success toast | **Raised by**: Reviewer B | **Dismissal Reason**: Low risk. Extra keys come from a `flightwall-settings.json` exported by the same device firmware, so types are always correct. A hand-crafted file with `timezone: 123` on a single-user LAN-only device with no authentication is not a threat model the story addresses. The existing `not null && not object` guard is sufficient.

- **Claimed Issue**: `processImportedSettings()` SRP violation — too many responsibilities in one function | **Raised by**: Reviewer C | **Dismissal Reason**: Pre-existing style, embedded JS. The function is 50 lines and straightforward. Splitting it would require closures or parameter passing that adds complexity without benefit in this context.

- **Claimed Issue**: DRY — error message string `'Could not read settings file — invalid format'` appears 3× | **Raised by**: Reviewer C | **Dismissal Reason**: Low-value refactor in embedded JS. Three occurrences in a LittleFS-deployed file; extracting a constant adds a variable declaration but saves 0 bytes after gzip. Deferred.

- **Claimed Issue**: Documentation gaps — WizardPage.ts not in story File List, Task 3 completion lie | **Raised by**: Reviewer C | **Dismissal Reason**: Out of scope. Story documentation auditing is not a source-code synthesis responsibility. The code defects underlying the Task 3 claim were addressed in rounds 1 and 2.
