## 🏛️ Architectural Sins

- **[6/10] Single Responsibility:** `tickNightScheduler()` is bolted directly into `main.cpp` rather than residing in a dedicated scheduler core component (e.g., `core/BrightnessScheduler.cpp`). This bloats the firmware entry point with display-policy logic and makes unit testing the scheduler in isolation impossible without spinning up the full device runtime.
  - 📍 `firmware/src/main.cpp:860-920`
  - 💡 Fix: Extract `BrightnessScheduler.h/.cpp` with `tick()` and `isDimming()` methods, initialized from `setup()` and ticked from `loop()`.

- **Boundary Breach:** The display task's config-change handler now juggles three unrelated concerns: hardware geometry changes, mapping reconfiguration, and brightness scheduling override. This handler has grown to ~45 lines and violates the Open/Closed principle—every new hot-reload feature will require touching this same block.
  - 📍 `firmware/src/main.cpp:324-367`

## 🐍 C++ Crimes & Readability

- **Missing explicit system header:** `main.cpp` uses `getLocalTime()`, `struct tm`, and `configTzTime()` but relies on transitive inclusion through `esp_sntp.h`. This is fragile—if the ESP-IDF header graph changes in a future framework update, compilation will break with crypto errors.
  - 📍 `firmware/src/main.cpp:1-40`
  - 💡 Fix: Add `#include <time.h>` explicitly.

## 🐛 Correctness & Safety

- **🐛 Bug — Test suite use-after-free:** `ConfigManager::init()` never clears `_callbacks`. Tests like `test_on_change_callback_fires` register lambdas capturing local `bool` variables by reference. When the test returns, those stack variables are freed. Later tests (e.g., `test_apply_json_schedule_hot_reload`) call `applyJson()` → `fireCallbacks()`, which invokes the stale lambdas, producing **undefined behavior** (writes to freed stack memory).
  - 📍 `firmware/core/ConfigManager.cpp:172-190` (missing `_callbacks.clear()`)
  - 🔄 Reproduction: Run the full test suite; `test_apply_json_schedule_hot_reload` will invoke stale callbacks from `test_on_change_callback_fires`.

- **🐛 Bug — AC #8 mismatch:** The acceptance criterion explicitly states: "`schedule_active` field reflects whether the scheduler is currently overriding brightness (not just whether it is enabled)". The implementation returns `(sched_enabled == 1) && isNtpSynced()`—this is **capability state**, not override state. The AC was not satisfied; it was intentionally contradicted by the Dev Notes.
  - 📍 `firmware/adapters/WebPortal.cpp:791`
  - 🔄 Reproduction: `GET /api/status` while in the dim window. `schedule_active` remains `true` even if `sched_enabled` is toggled off, as long as NTP is synced.

- **🐛 Bug — Dead code:** `g_ntpSynced` is set to `true` exclusively inside `onNtpSync()` and is **never reset to `false`** anywhere in the codebase. Consequently, the branch `!ntpSynced && g_lastNtpSynced` inside `tickNightScheduler()` is permanently unreachable dead code that misleads maintainers into believing NTP-loss recovery exists.
  - 📍 `firmware/src/main.cpp:890-893`

- **🐛 Bug — Unrepaired NVS corruption:** `ConfigManager::loadFromNvs()` detects corrupted schedule values (e.g., `sched_dim_start = 65000`) and clamps them to safe defaults while logging a warning. However, it **never writes the clamped value back to NVS**, so the warning repeats on every boot until the user manually overwrites the key.
  - 📍 `firmware/core/ConfigManager.cpp:245-258`

- **🐛 Bug — Missing defensive time bounds:** `tickNightScheduler()` computes `nowMinutes = (uint16_t)(timeinfo.tm_hour * 60 + timeinfo.tm_min)` immediately after `getLocalTime(&timeinfo, 0)` without validating that `tm_hour ∈ [0,23]` and `tm_min ∈ [0,59]`. On a 24/7 embedded device, a corrupted RTC or newlib edge case could produce out-of-range values, causing the scheduler to make incorrect dimming decisions for up to a day.
  - 📍 `firmware/src/main.cpp:905-906`
