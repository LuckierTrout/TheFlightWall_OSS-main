## 🏛️ Architectural Sins

- **[3/10] Single Responsibility (Consistency):** The codebase establishes a clean event delegation pattern for keyboard events on the mode picker (`modeCardsList.addEventListener('keydown', ...)`), but ignores this pattern for click events, instead attaching an inline `addEventListener('click')` closure to every single card in a loop during `renderModeCards`. This wastes memory and breaks architectural consistency.
  - 📍 `firmware/data-src/dashboard.js:1500`
  - 💡 Fix: Remove the loop-bound click listener and implement event delegation on `modeCardsList` for clicks, mirroring the `keydown` block.

---

## 🐍 Pythonic Crimes & Readability

- **Style/Readability:** Hardcoded structural text in HTML causes visual artifacts. `dashboard.html` hardcodes parentheses around the `modeStatusReason` span. When `data.state_reason` is empty, `dashboard.js` sets the span text to `''`, leaving a dangling empty set of parentheses `Active: Classic Card ()` on the UI.
  - 📍 `firmware/data-src/dashboard.html:79` and `dashboard.js:1396`

## ⚡ Performance & Scalability

- **[Medium] UI Layout Shift:** When a card transitions to `.active`, `style.css` changes `border: 1px` to `border-left: 4px`. Because `box-sizing: border-box` is used, this 3px difference shrinks the internal content box, causing the text and schematic to jitter/shift 3px to the right. This is a classic UI performance footgun (Cumulative Layout Shift).
  - 📍 `firmware/data-src/style.css:266`
  - 💡 Fix: Add `padding-left: 9px;` (reduced from 12px) to `.mode-card.active` to mathematically compensate for the +3px border width.

## 🐛 Correctness & Safety

- **🐛 Bug (CRITICAL):** Scoped error messages are instantly obliterated. In `switchMode()`, when an error occurs, `showCardError(targetCard, msg)` is called. However, immediately after, `loadDisplayModes()` (or `renderModeCards(d)` synchronously) is invoked. This destroys `modeCardsList.innerHTML` and replaces all cards, instantly garbage-collecting the newly attached error alert before the browser even paints it. The user has no idea why the switch failed.
  - 📍 `firmware/data-src/dashboard.js:1551`
  - 🔄 Reproduction: Simulate an HTTP error or polling mismatch. The card flashes "Switching..." and then instantly returns to normal with no error toast or inline message.

- **🐛 Bug (IMPORTANT):** `buildSchematic` creates invalid CSS if `z.relX` or `z.relY` are missing from the API payload. While `relW`/`relH` have a `continue` guard, `Math.max(1, z.relX + 1)` evaluates to `NaN` if `z.relX` is `undefined`. This injects `grid-column: NaN / NaN` into the DOM, breaking the schematic layout entirely.
  - 📍 `firmware/data-src/dashboard.js:1449`
  - 🔄 Reproduction: Provide an API response where `relX` or `relY` is omitted (a common JSON serialization optimization for `0`).

- **🐛 Bug (IMPORTANT):** `switchMode` polling uses `Date.now() - pollStart > SWITCH_POLL_TIMEOUT`. Modern browsers throttle `setTimeout` to >= 1000ms in background tabs. If a user clicks a mode and immediately switches tabs, the timeout executes just twice but exceeds 2000ms of real-world time, falsely rejecting a successful hardware switch.
  - 📍 `firmware/data-src/dashboard.js:1578`
  - 🔄 Reproduction: Click "Switch" and instantly switch browser tabs. Return 3 seconds later. The UI shows "Timed out" even though the physical LED wall switched successfully.
