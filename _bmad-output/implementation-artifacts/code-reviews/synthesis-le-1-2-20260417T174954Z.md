The story Dev Notes explicitly says "grep-only." No compile-time test is required; the grep gate is valid.

- **Claimed Issue**: `WidgetType` enum bounds validation — `dispatch()` silently ignores `static_cast<WidgetType>(99)` | **Raised by**: Reviewer C | **Dismissal Reason**: The switch has explicit cases for all 6 enum values including `WidgetType::Unknown`. The trailing `return false` on line 62 catches any out-of-range cast. `-Wswitch` without a default on the known-type cases provides compile-time coverage when new enum values are added. Adding a runtime bounds check would add code to the hot path for a scenario only reachable via intentional API misuse (invalid cast), which is not a defensible production failure mode.

- **Claimed Issue**: WidgetSpec.content not guaranteed null-terminated | **Raised by**: Reviewer C | **Dismissal Reason**: `makeSpec()` in the test helper uses `strncpy(s.content, content, sizeof(s.content) - 1)` and the struct is zero-initialised (`WidgetSpec s{}`), guaranteeing null termination. LE-1.3 (the production caller) is responsible for null-terminating the field after JSON parse — this is standard C++ practice. The render functions' `content[0] == '\0'` guard is a correct first-character check.

- **Claimed Issue**: LogoWidget static buffer leaks previous widget's color if spec is undersized | **Raised by**: Reviewer C | **Dismissal Reason**: `renderLogo()` guards `(int)spec.w < 8 || (int)spec.h < 8 → return true` *before* any buffer fill. An undersized spec never reaches the `s_buf` fill loop or `drawBitmapRGB565`. The buffer is only filled when the function will draw, so there is no color bleed. (The reviewer's claim about "widget 4 reads stale buffer from widget 1" cannot occur because the fill always precedes the draw in the same call.)

- **Claimed Issue**: Inconsistent guard order across widget render functions (matrix check vs dimension check order) | **Raised by**: Reviewer C | **Dismissal Reason**: Verified in source. Both `TextWidget.cpp` and `ClockWidget.cpp` check dimensions first then `ctx.matrix == nullptr`; `LogoWidget.cpp` checks dimensions first then matrix. The order is intentional: undersized dimension short-circuits before any other work regardless of hardware state. The `ClockWidget` does the cache refresh (pure in-memory work, no matrix required) *before* the matrix guard — this is correct because the cache must remain hot for the frame even when `ctx.matrix == nullptr` in tests.

- **Claimed Issue**: Multiple clock widgets in one layout share global static cache (LE-1.3 collision) | **Raised by**: Reviewer A | **Dismissal Reason**: Correctly flagged as a *future* LE-1.3 concern but not a current defect. LE-1.2 scope is strictly additive. The story Dev Notes explicitly defer multi-clock handling: "If multiple clocks are allowed, remove global static cache from `ClockWidget.cpp` in that story [LE-1.3]." This is a known acceptable limitation documented by design.

- **Claimed Issue**: `isKnownType("flight_field"|"metric")` returns true but `dispatch` returns false — semantic mismatch | **Raised by**: Reviewer A | **Dismissal Reason**: The contract is deliberate and tested explicitly (lines 90–103 in test_main.cpp). `isKnownType` means "is this a valid type string from the JSON schema" — used by LayoutStore validation. `dispatch` returning `false` for FlightField/Metric means "not yet renderable" (LE-1.8 scope). ## Changes Applied

**File**: `firmware/widgets/ClockWidget.cpp`
**Change**: Fix NTP-down hot-path runaway — advance `s_lastClockUpdateMs` regardless of `getLocalTime()` success/failure. Also updated file-header doc comment to describe the corrected behaviour.

**Before**:
```cpp
        if (getLocalTime(&tinfo, 0)) {
            snprintf(s_clockBuf, ...);
            s_lastClockUpdateMs = nowMs;
            s_timeCallCount++;
        }
        // On failure: leave s_clockBuf / s_lastClockUpdateMs alone so the
        // next render retries rather than waiting the full 30s window.
```
**After**:
```cpp
        if (getLocalTime(&tinfo, 0)) {
            snprintf(s_clockBuf, ...);
            s_timeCallCount++;
        }
        // Advance the timestamp regardless of success or failure so that
        // failed attempts are also throttled to ~30s. ...
        s_lastClockUpdateMs = nowMs;
```
