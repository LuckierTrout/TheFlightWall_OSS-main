**Findings**

1. **High:** AC #7 is no longer implemented for most failure paths. The banner’s primary CTA only calls `FW.showToast('Mode not available', 'error')` when `live_flight` is absent from the last GET payload; POST `ok:false`, `registry_error`, and timeout failures are delegated to `switchMode()`, which surfaces inline card errors instead of the required toast. That is a direct acceptance-criteria regression against [the story AC]( /Users/christianlee/App-Development/TheFlightWall_OSS-main/_bmad-output/implementation-artifacts/stories/ds-3-6-upgrade-notification-banner.md:31 ) and the current `tryBtn` handler in [dashboard.js](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/dashboard.js:3160) plus the `switchMode()` failure branches in [dashboard.js](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/dashboard.js:2890).

2. **High:** Clicking “Try Live Flight Card” during an existing mode transition silently discards the one-time banner. The handler runs `dismissAndClearUpgrade()` before calling `switchMode('live_flight')`, but `switchMode()` immediately returns when `modeSwitchInFlight` is already true, so the user gets no switch, no error, and no banner retry. That is a real state-loss bug in [dashboard.js](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/dashboard.js:3168) and [dashboard.js](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/dashboard.js:2890).

3. **Medium:** The dismiss pipeline treats firmware-side dismissal failure as success. `dismissAndClearUpgrade()` sets local storage, removes the DOM node, and swallows any `POST /api/display/notification/dismiss` error. If that POST fails, `upg_notif` remains set in NVS, so the banner reappears on another browser or after storage is cleared. That breaks the intended dual-source “one-time” behavior in [dashboard.js](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/dashboard.js:3131).

4. **High:** The E2E harness is exercising the wrong `/api/display/modes` contract. The dashboard code consumes `data.active`, `data.state_reason`, and `mode.zone_layout` in [dashboard.js](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/dashboard.js:2728), but the test fixtures and mock server emit `active_mode`, omit `state_reason`, and never provide `zone_layout` in [api-responses.ts](/Users/christianlee/App-Development/TheFlightWall_OSS-main/tests/e2e/fixtures/api-responses.ts:92), [api-helpers.ts](/Users/christianlee/App-Development/TheFlightWall_OSS-main/tests/e2e/helpers/api-helpers.ts:235), and [server.ts](/Users/christianlee/App-Development/TheFlightWall_OSS-main/tests/e2e/mock-server/server.ts:377). These tests can pass while the real dashboard integration is broken.

5. **Medium:** The “comprehensive E2E coverage” claim is overstated. The new suite never asserts the required toast for AC #7, never simulates the timeout path it claims to cover, and mostly stops after checking that the banner disappeared. Because of the API-shape drift above, even the supposed success path is not proving that the CTA actually completed a valid mode switch. See [upgrade-banner.spec.ts](/Users/christianlee/App-Development/TheFlightWall_OSS-main/tests/e2e/tests/upgrade-banner.spec.ts:193) and [upgrade-banner.spec.ts](/Users/christianlee/App-Development/TheFlightWall_OSS-main/tests/e2e/tests/upgrade-banner.spec.ts:286).

6. **Low:** The banner is injected asynchronously after the first modes fetch but is not marked live. `role="region"` plus `aria-labelledby` names it, but many screen readers will not announce a newly inserted informational region unless it is also live. The story explicitly allowed `aria-live="polite"` for this case, and the omission weakens discoverability in [dashboard.js](/Users/christianlee/App-Development/TheFlightWall_OSS-main/firmware/data-src/dashboard.js:3147).

**Suggested Fixes**

1. Make the banner CTA preserve failure semantics explicitly: if `modeSwitchInFlight` is true, do not dismiss yet; otherwise await the `switchMode()` outcome and emit the AC #7 toast for every failed primary-CTA path, not just the “mode missing” branch.

2. Stop swallowing dismiss failures. Either await the dismiss POST before removing the banner, or record a pending retry so the firmware-side `upg_notif` flag is eventually cleared.

3. Fix the test doubles first. Align `tests/e2e` and the mock server to the real `GET /api/display/modes` payload, then add assertions for timeout, `ok:false`, `registry_error`, and the required user-visible feedback.
