## Issues Verified (by severity)

### Critical

1. **Architecture Rule 18 Violation: NVS flash read on Core 0 display task hot path**
   - **Source**: Reviewer B
   - **File**: `firmware/modes/ClockMode.cpp:85`
   - **Evidence**: `render()` method calls `ConfigManager::getModeSetting("clock", "format", 0)` every time the second changes. Project architecture rule 18 explicitly states: "Modes must ONLY access data through RenderContext — zero ConfigManager reads from inside a mode" and "Modes → ConfigManager | Via init() only"
   - **Fix**: Removed line 85 (`_format = (uint8_t)ConfigManager::getModeSetting("clock", "format", 0);`). The format setting is already cached in `_format` member variable during `init()` (line 42). Mode now relies exclusively on cached value.

2. **Open/Closed Principle (SOLID) Violation: Hardcoded mode validation in ConfigManager**
   - **Source**: Reviewer B
   - **File**: `firmware/core/ConfigManager.cpp:787-792`
   - **Evidence**: `setModeSetting()` contains hardcoded validation logic for specific modes:
     ```cpp
     if (strcmp(abbrev, "clock") == 0 && strcmp(key, "format") == 0) {
         if (value < 0 || value > 1) return false;
     }
     ```
     This defeats the purpose of the dynamic `ModeSettingsSchema` architecture and forces ConfigManager to be modified for every new mode. Story task 1.2 incorrectly required this pattern.
   - **Fix**: Removed hardcoded validation blocks (lines 787-792). Added comment explaining validation should occur at API layer using `ModeSettingsSchema` (minValue/maxValue fields) before calling `setModeSetting`. ConfigManager is now storage-only.

### High

3. **UI Bouncing Bug: Dynamic text scaling causes jarring layout jumps twice daily**
   - **Source**: Reviewer B
   - **File**: `firmware/modes/ClockMode.cpp:91` (originally line 94)
   - **Evidence**: 12-hour format uses `%d:%02d%s` which produces variable-length strings:
     - "9:59AM" = 6 characters → scale factor 4
     - "10:00AM" = 7 characters → scale factor 3
     
     Dynamic text scaling (lines 128-135) recalculates size based on string length, causing the clock to suddenly shrink and reposition at 10:00 AM and 10:00 PM.
   - **Reproduction**: Set clock format to 12-hour, observe transition from 9:59 AM to 10:00 AM on LED matrix
   - **Fix**: Changed format specifier from `%d` to `%2d` to pad single-digit hours with leading space. String length now constant at 7 characters for all 12-hour times (e.g., " 9:30AM", "10:30AM").

4. **Lying Test: Enforces architectural anti-pattern**
   - **Source**: Reviewer B
   - **File**: `firmware/test/test_config_manager/test_main.cpp:653-661`
   - **Evidence**: `test_set_mode_setting_validates_clock_format_range` verifies that ConfigManager rejects out-of-range values:
     ```cpp
     TEST_ASSERT_FALSE(ConfigManager::setModeSetting("clock", "format", 2));
     ```
     This test actively enforces the OCP violation identified in Issue #2. It locks in bad design by requiring ConfigManager to know about mode internals.
   - **Fix**: Updated test expectations to match storage-only architecture. ConfigManager now accepts all values; test verifies that out-of-range values ARE accepted (API layer will validate). Updated test comments to explain architectural decision.

### Medium

No medium severity issues identified.

### Low

5. **Tech Debt: ODR violation and flash bloat from static const in header**
   - **Source**: Reviewer B
   - **File**: `firmware/modes/ClockMode.h:17-25`
   - **Evidence**: `CLOCK_SETTINGS` and `CLOCK_SCHEMA` declared as `static const` arrays directly in header file. Every translation unit that includes this header allocates its own copy in flash memory, causing firmware bloat and potential pointer-inequality bugs if schemas are ever compared by reference.
