## Issues Dismissed

- **Claimed Issue**: `applyWithReboot()` used for Night Mode keys instead of `applySettings()` | **Raised by**: Reviewer D | **Dismissal Reason**: The unified Apply button routes all sections through `applyWithReboot()` — this is the pre-existing architecture. The server returns `reboot_required: false` for schedule keys; the function handles both cases correctly. Fixing this would require restructuring the entire apply flow. Previously adjudicated fn-2.3 antipatterns.

- **Claimed Issue**: `TZ_MAP` duplicated in `dashboard.js` and `wizard.js` | **Raised by**: Reviewers A, B | **Dismissal Reason**: Explicitly adjudicated as deferred in fn-2.1 antipatterns (Rounds 1–3). Moving to `common.js` requires wizard page script-load validation; "no new files" constraint blocked action.

- **Claimed Issue**: `posixToIana()` UTC fallback silently shows wrong timezone for custom POSIX strings | **Raised by**: Reviewer B | **Dismissal Reason**: The `nmTimezoneDirty` flag (already in place) prevents silent data overwrite — the user must explicitly change the dropdown to corrupt a custom POSIX value. The display being "UTC" for an unsupported custom string is a UX concern but adding a "Custom" dropdown option is complex and out of scope for synthesis.

- **Claimed Issue**: `/api/status` missing fields treated as falsy operational states | **Raised by**: Reviewer B | **Dismissal Reason**: The response shape is tightly controlled by the device firmware (fn-2.2). The firmware always serializes these fields. "Unknown/unsupported firmware" detection would require version-sniffing logic not warranted for this embedded single-device scenario.

- **Claimed Issue**: `aria-live` region rewritten every 10s causing screen-reader noise | **Raised by**: Reviewer B | **Dismissal Reason**: Minor UX concern. A state-diff guard would add complexity; the polling writes identical text strings most of the time, and many screen reader implementations deduplicate identical text announcements.

- **Claimed Issue**: Redundant `/api/status` calls from `loadFirmwareStatus()` and `updateNmStatus()` on page load | **Raised by**: Reviewers A, D | **Dismissal Reason**: `loadFirmwareStatus()` predates fn-2.3 and is a one-time call. On the ESP32 (2-3 concurrent client limit), two simultaneous requests on load are within capacity. A shared status manager would be a larger refactor outside story scope.

- **Claimed Issue**: `setTimeout` poll timer never cleared on `beforeunload` | **Raised by**: Reviewer D | **Dismissal Reason**: The dashboard serves a single embedded device; page navigation away is rare. No persistent state is leaked (the timer fires to a device that may no longer be the open tab). Negligible real-world impact.

- **Claimed Issue**: SOLID/SRP violations in `collectPayload()` / `loadSettings()` God Functions | **Raised by**: Reviewers A, B | **Dismissal Reason**: Pre-existing pattern across the entire dashboard predating fn-2.3. A registry/controller refactor is a dedicated future story concern.

- **Claimed Issue**: Missing automated regression tests for `timeToMinutes()`, `posixToIana()`, etc. | **Raised by**: Reviewer B | **Dismissal Reason**: No JS test infrastructure exists in this project (firmware-only Unity tests). All prior story reviews reached the same conclusion. Actionable only as a future story to add host-side JS tests.

- **Claimed Issue**: `.gz` rebuild could not be verified | **Raised by**: Reviewer D | **Dismissal Reason**: Both `.gz` files were rebuilt in this synthesis pass and verified with `gzip -t`.
