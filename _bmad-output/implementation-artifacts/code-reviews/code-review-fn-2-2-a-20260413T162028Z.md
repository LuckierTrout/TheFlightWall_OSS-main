## Findings

1. `schedule_active` is still wrong. AC #8 says it must reflect whether the scheduler is actively overriding brightness, but `_handleGetStatus()` returns `(ConfigManager::getSchedule().sched_enabled == 1) && isNtpSynced()` instead. That is just “scheduler can run,” not “scheduler is dimming now.” `schedule_dimming` was added, but the original contract is still unmet. See `firmware/adapters/WebPortal.cpp:788-791` against story `:62-66`.

2. AC #6 is not what the code does. The story and Architecture F5 say a manual brightness change during the dim window should apply, then get re-overridden on the next scheduler tick. The implementation applies `localDisp.brightness` and immediately overwrites it with `sched_dim_brt` in the same `displayTask()` branch, so there is no meaningful “momentary” manual brightness and no one-second grace period. See `firmware/src/main.cpp:359-369` against story `:49-54`.

3. The scheduler is not actually non-blocking. `tickNightScheduler()` calls `ConfigManager::getSchedule()` on every run, and that getter acquires `g_configMutex` with `xSemaphoreTake(..., portMAX_DELAY)`. The same mutex is held across the full `persistToNvs()` flash write path. Under a pending persist or concurrent config write, the scheduler can block the Core 1 loop and delay WiFi, OTA self-check, and fetch work, which is the opposite of AC #1. See `firmware/src/main.cpp:890-895` and `firmware/core/ConfigManager.cpp:35-40`, `438-486`, `578-580`.

4. The tests are too weak for the regressions this story introduced. The new tests only cover `isInDimWindow()` arithmetic and defaults. They do not validate `schedule_active`/`schedule_dimming`, do not validate `g_scheduleChanged` signaling, do not validate restore-on-disable, and do not validate the manual-brightness timing contract. That is exactly why the broken `/api/status` semantics shipped unnoticed. See `firmware/test/test_config_manager/test_main.cpp:528-551` and `569-620`.

5. The scheduler’s NTP transition log is misleading. At `g_ntpSynced` transition, it logs `"NTP synced — schedule active"` before it checks `sched_enabled`. If night mode is disabled, the device emits a false “active” message and then immediately returns through the disabled guard. That makes field diagnostics worse, not better. See `firmware/src/main.cpp:899-914`.

## Architectural Sins

- **[7/10] Dependency inversion / boundary breach:** the hot-path scheduler depends on a blocking, flash-coupled config access path instead of cached runtime state.
  - 📍 `firmware/src/main.cpp:890-895`
  - 📍 `firmware/core/ConfigManager.cpp:35-40`, `438-486`, `578-580`
  - Fix: cache the four scheduler primitives (`enabled`, `dim_start`, `dim_end`, `dim_brt`) in `main.cpp` via the existing config-change callback, or expose a lock-free snapshot dedicated to the scheduler hot path.

## Performance & Scalability

- **[Medium] Blocking hot path:** the scheduler’s once-per-second tick can wait indefinitely on `ConfigManager`’s mutex while NVS writes are in progress.
  - 📍 `firmware/src/main.cpp:890-895`
  - 📍 `firmware/core/ConfigManager.cpp:35-40`, `438-486`
  - Fix: stop taking a blocking mutex from the scheduler path.

## Correctness & Safety

  - 📍 `firmware/adapters/WebPortal.cpp:788-791`
  - Reproduction: enable schedule, stay outside the dim window, call `/api/status`; `schedule_active` still returns `true`.

- **Bug:** manual brightness change during dim window never survives to the next scheduler tick.
  - 📍 `firmware/src/main.cpp:359-369`
  - Reproduction: enable dim window, move brightness slider; the display task reapplies `sched_dim_brt` immediately in the same handler.

- **Lying / insufficient test:** scheduler tests do not exercise runtime behavior or API contract.
  - 📍 `firmware/test/test_config_manager/test_main.cpp:569-620`
  - Why it lies: it proves only the extracted arithmetic helper, not the story’s actual runtime contracts.

## Suggested Fixes

1. Change `/api/status` so `schedule_active` reflects current dimming state, or rename the current field to `schedule_enabled` and keep `schedule_dimming` as the real-time flag. Right now the API contract is split-brain.

2. Decide the intended AC #6 behavior and implement it consistently. If Architecture F5 is authoritative, remove the immediate re-override from `displayTask()` and let the next scheduler tick reclaim brightness. If immediate re-override is truly desired, then the story/AC must be corrected before approval.

3. Remove blocking `ConfigManager::getSchedule()` from `tickNightScheduler()`. Cache scheduler parameters in main-loop state from the existing config callback instead of taking a `portMAX_DELAY` mutex in the hot path.

4. Add at least one host-side/component test for `/api/status` scheduler fields and one runtime test for the dimming state machine. The current Unity additions are arithmetic smoke tests, not acceptance coverage.

5. Fix the log message so it says `"NTP synced"` or `"NTP synced — scheduler available"` unless `sched_enabled == 1`.
