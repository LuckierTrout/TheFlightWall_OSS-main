This bypasses the `SystemStatus` abstraction, which is intended to consolidate system health and status information. If `ota_rb_ack` is a system status, it should be part of `SystemStatus`'s model and exposed via `SystemStatus::toExtendedJson()`, ensuring a single source of truth and consistent access pattern.
  - 📍 `firmware/adapters/WebPortal.cpp:896`
  - 💡 Fix: Extend `SystemStatus` to manage and expose the `ota_rb_ack` state, removing direct NVS access from `WebPortal`.

## ⚡ Performance &amp; Scalability

- **[Medium] Frequent DOM Manipulation:** In `dashboard.js`, the `updateOtaProgress` function is called directly within the `xhr.upload.onprogress` event handler. While `XMLHttpRequest` progress events are generally throttled by the browser, rapid events (e.g., on a fast local network) could lead to frequent DOM updates (`style.width`, `setAttribute`, `textContent`) potentially causing minor layout thrashing or unnecessary re-renders.
  - 📍 `firmware/data-src/dashboard.js:2708`
  - 💡 Fix: Implement a simple throttling mechanism (e.g., `requestAnimationFrame` or a `setTimeout` with a short delay) to limit updates to a reasonable frequency (e.g., every 100-200ms or on next animation frame), ensuring smooth animation without over-taxing the browser.

## 🐛 Correctness &amp; Safety

- **🐛 Bug:** In `dashboard.js`, the `startRebootPolling` function polls `/api/status` every 3 seconds for up to 60 seconds. While the `?_=` cache-busting parameter and `done` guard mitigate some race conditions, there's a subtle scenario: if the device reboots very quickly (e.g., < 3s after countdown) and `loadFirmwareStatus()` (called on page load) fetches a stale `firmware_version` before the `startRebootPolling` can get a fresh one, the UI might briefly show the old version before being updated by the polling mechanism. This is a minor display glitch but indicates a potential for temporary inconsistency.
  - 📍 `firmware/data-src/dashboard.js:2743`
  - 🔄 Reproduction: Perform a very fast OTA update on a device that reboots extremely quickly (unlikely but possible). Observe the `fw-version` text. It *could* briefly flicker back to the old version before the polling mechanism updates it.

- **🔒 [Medium] Security:** The `POST /api/ota/ack-rollback` endpoint in `WebPortal.cpp` (and other API endpoints like `/api/reboot`, `/api/reset`) lack any form of authentication or CSRF protection. While this is explicitly dismissed in previous anti-patterns as "LAN-only device" and "requires a dedicated security story," its absence on a state-modifying endpoint remains a security vulnerability. An attacker on the local network could easily send this request to dismiss critical warnings, potentially misleading a user about device state.
  - 📍 `firmware/adapters/WebPortal.cpp:542`
  - ⚠️ Impact: An attacker on the local network could maliciously dismiss important security/device state warnings, compromising user trust and device integrity.
