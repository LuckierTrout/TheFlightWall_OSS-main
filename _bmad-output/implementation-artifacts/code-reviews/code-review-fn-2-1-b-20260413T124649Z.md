## Findings

1. **🟠 Important: the “fixed” timezone hot-reload guard still restarts SNTP on the first unrelated config change after boot.**  
   `firmware/src/main.cpp:659` initializes `s_lastAppliedTz` as empty and only seeds it inside the callback. Boot-time sync happens separately in `firmware/adapters/WiFiManager.cpp:132-136`, so the first hot-reload of an unrelated key like `brightness` still satisfies `sched.timezone != s_lastAppliedTz` and calls `configTzTime()` again. The story claims this was fixed at `_bmad-output/.../fn-2-1-ntp-time-sync-and-timezone-configuration.md:324-330`, but it is only partially fixed.

2. **🟠 Important: `ConfigManager` accepts an empty timezone string and pushes it straight into `configTzTime()`.**  
   `firmware/core/ConfigManager.cpp:135-140` validates only type and max length; `""` is accepted. Both `firmware/adapters/WiFiManager.cpp:133-136` and `firmware/src/main.cpp:660-663` then pass that value to `configTzTime()`. The test suite checks “too long” values only (`firmware/test/test_config_manager/test_main.cpp:366-382`), so this bad input path is unguarded.

3. **🟠 Important: `ntp_synced` becomes a sticky false-positive after the first successful sync.**  
   `firmware/src/main.cpp:75-81` only ever sets `g_ntpSynced` to `true`. Nothing clears it on WiFi loss (`firmware/adapters/WiFiManager.cpp:144-159`), SNTP reset, or timezone reconfiguration. `firmware/adapters/WebPortal.cpp:786-787` then exposes that stale bit as `ntp_synced` and uses it to derive `schedule_active`. After one good sync, the status endpoint can keep claiming the clock is synced even when the device is offline.

4. **🟠 Important: AC #2 is only half-implemented; there is no verification that local time is actually usable after sync.**  
   The acceptance criterion requires `getLocalTime(&now, 0)` to return correct local time. The production callback in `firmware/src/main.cpp:75-81` only flips a flag and status; there is no `getLocalTime` validation path in the reviewed implementation, and no test covers it. The story’s “all 6 acceptance criteria satisfied” claim at `_bmad-output/.../fn-2-1-ntp-time-sync-and-timezone-configuration.md:319-326` is too strong.

5. **🟠 Important: `test_ntp_status_in_json_output` is a lying test for AC #6.**  
   `firmware/test/test_config_manager/test_main.cpp:522-543` exercises `SystemStatus::toJson()`, not `WebPortal::_handleGetStatus()`. The actual endpoint fields added by this story are in `firmware/adapters/WebPortal.cpp:777-787`. A regression that drops `ntp_synced` or `schedule_active` from `/api/status` would still pass this test suite.

6. **🟠 Important: `schedule_active` is semantically wrong.**  
   `firmware/adapters/WebPortal.cpp:787` reports `schedule_active = sched_enabled && isNtpSynced()`. That is not “active”; it is “feature enabled and clock bit is true.” Any client reading this as “currently inside the dim window” will be misled. This is already API debt before the scheduler lands.

7. **🟡 Minor: the timezone helper silently maps unsupported browser zones to UTC, and the map is duplicated in two bundles.**  
   `firmware/data-src/dashboard.js:94-102` and `firmware/data-src/wizard.js:93-99` both do `TZ_MAP[iana] || "UTC0"`. Common browser zones such as `America/Detroit`, `Asia/Manila`, and `Europe/Prague` are absent, so unsupported users silently get the wrong timezone instead of an actionable error. The full `TZ_MAP` is also copy-pasted in both files (`dashboard.js:8-87`, `wizard.js:8-87`), so every DST fix now requires dual edits and dual gzip rebuilds.

8. **🟡 Minor: the story’s file accounting no longer matches git reality.**  
   The story says eight product files were modified (`_bmad-output/.../fn-2-1-ntp-time-sync-and-timezone-configuration.md:342-351`), but `git diff HEAD -- ...` currently shows only `firmware/src/main.cpp` as dirty among those product paths. That makes the story’s “File List” and completion notes non-reproducible from the current worktree.
