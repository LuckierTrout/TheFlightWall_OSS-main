- **Issue**: `displayFallbackCard` calls `renderZoneFlight` when `_layout.valid`, violating AC #1 ("reuse `displaySingleFlightCard` / `displayLoadingScreen` draw logic") and D3 (NeoMatrixDisplay owns hardware and frame commit only) | **Source**: Validator B | **File**: `firmware/adapters/NeoMatrixDisplay.cpp` | **Fix**: Removed `if (_layout.valid) { renderZoneFlight(flights[0]); } else { ... }` conditional — fallback now unconditionally calls `displaySingleFlightCard(flights[0])` per AC #1.

### Low

- **Issue**: `millis()` overflow in status message expiry calculation — `statusMessageUntilMs = millis() + statusMessage.durationMs` produces an absolute deadline that overflows to a small value at 49-day uptime; `millis() < statusMessageUntilMs` then immediately evaluates false, causing messages to flash for one frame | **Source**: Validator B | **File**: `firmware/src/main.cpp` | **Fix**: Replaced single `statusMessageUntilMs` variable with `statusMessageStartMs` (time queued) + `statusMessageDurationMs` (duration). Expiry check now uses overflow-safe unsigned subtraction: `(millis() - statusMessageStartMs) < statusMessageDurationMs`.

## Issues Dismissed

- **Claimed Issue**: Fallback card "permanently loses flight cycling logic" — users stare at a single flight forever during mode failure | **Raised by**: Validator B | **Dismissal Reason**: By design per AC #5. The fallback's purpose is to ensure "the wall never stays blank while flight data exists" — not to cycle flights. Flight cycling is ClassicCardMode's responsibility (AC #3, ds-1.4). The fallback is an emergency path, not a cycling display. Rendering `flights[0]` is the correct and intended behavior when the mode system is unavailable.

- **Claimed Issue**: Magic numbers `2000` and `4000` in `tickStartupProgress` startup state machine | **Raised by**: Validator B | **Dismissal Reason**: These hardcoded timing literals are self-documenting in context (surrounding comments explain "Show WiFi Connected", "Show IP address", "Advance to authentication"), follow standard embedded firmware conventions, and appear in a non-hot path that's only exercised once at boot. The existing `ipShown` guard (added in ds-1.5 synthesis) already addresses the actual spam bug. Extracting named constants would provide no bug-prevention value here.
