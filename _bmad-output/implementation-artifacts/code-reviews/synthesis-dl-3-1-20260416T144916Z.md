## Issues Dismissed

- **Reviewer A: Stale Render State on Zero-Dimension Skip** - Dismissed as negligible impact
  - **Claimed Issue**: When fade is skipped due to zero dimensions, new mode doesn't render before returning to IDLE, creating a "gap" where mode is active but hasn't rendered yet.
  - **Raised by**: Reviewer A
  - **Dismissal Reason**: Display task runs at ~50Hz (20ms period). The "gap" lasts a single tick (≤20ms) before the next render. The mode's `init()` has succeeded and the mode is ready to render. This is consistent with the existing instant-handoff behavior when fade allocation fails - the new mode becomes active and renders on the next orchestrator tick. AC #1 requires the mode to be "ready to take over" (init complete), not "has already rendered." The edge case is cosmetic and doesn't violate any AC.

- **Reviewer A: test_fade_blend_math_correctness doesn't test actual fade loop** - Dismissed as intentional design
  - **Claimed Issue**: Test verifies lerp formula in isolation but doesn't integrate with actual fade loop, so bugs in blend loop wouldn't be caught.
  - **Raised by**: Reviewer A
  - **Dismissal Reason**: This is a unit test for mathematical correctness of the lerp algorithm, not an integration test. The actual fade loop is tested indirectly by tests like `test_switch_with_zero_layout_no_crash` and `test_rapid_switches_with_fade_no_crash` which verify the entire switch path completes without crash. Full integration testing of the 15-step crossfade requires physical LED hardware and can't be done in the native test environment. The unit test provides appropriate coverage for the math, and the no-crash tests verify the integration path doesn't break.

- **Reviewer B: SRP Violation - ModeRegistry handles registration AND fade animation** - Dismissed as misunderstanding of responsibility boundaries
  - **Claimed Issue**: ModeRegistry violates Single Responsibility Principle by handling both mode registration and fade transition logic.
  - **Raised by**: Reviewer B
  - **Dismissal Reason**: Per Architecture Decision D2, ModeRegistry's responsibility is "cooperative switch serialization" - coordinating the entire mode lifecycle on Core 0. Fade transition is part of the switch lifecycle (occurs between init and IDLE state return). Extracting fade to a separate "TransitionEngine" would fragment the switch sequence across multiple components and complicate buffer lifetime management. The fade logic is ~60 lines in a 546-line file - not a "God Object." This is appropriate cohesion for embedded systems where minimizing function call overhead matters.

- **Reviewer B: Multiple show() calls violate NFR architecture** - Dismissed as already documented exception
  - **Claimed Issue**: AC #9 documents multiple FastLED.show() calls during transition, but this violates FR35 single-show-per-frame rule.
  - **Raised by**: Reviewer B
  - **Dismissal Reason**: AC #9 explicitly documents this as an "explicit exception to FR35 single-show-per-frame rule." The story already acknowledges and documents this architectural tradeoff. The SWITCHING state prevents orchestrator/HTTP interference during the ~1s transition window. This is not a bug - it's a documented design decision.

- **Reviewer B: Incomplete init error handling with buffer leak** - Dismissed as incorrect
  - **Claimed Issue**: If init() fails during switch after fade buffer allocated, buffer is leaked if not carefully managed across all error branches.
  - **Raised by**: Reviewer B
  - **Dismissal Reason**: Code review confirms all early-exit paths in executeSwitch() free fadeSnapshotBuf: line 188 (heap guard), line 226 (factory null), line 268 (init fail). The allocation happens at line 120 and the buffer is freed at line 349 after fade completes OR on any early exit. No leak path exists.
