## Issues Dismissed

- **Claimed**: "No 2-second progress update guarantee" | **Raised by**: Reviewer A | **Reason**: Story Dev Notes explicitly state "add a minimum interval check if needed" — the qualifier "if needed" signals this was considered optional. On a local LAN upload, XHR `upload.onprogress` fires per-chunk far more frequently than every 2s. Adding an artificial `setInterval` would create visual noise on fast uploads. The AC is satisfied in practice; forcing a synthetic timer is over-engineering for an embedded LAN device.

- **Claimed**: "Zero-byte file passes validation and gets uploaded" | **Raised by**: Reviewer C | **Reason**: Logic error in review — `new Uint8Array(emptyArrayBuffer)[0]` returns `undefined`; `undefined !== 0xE9` evaluates to `true`, so the code enters the error branch and shows "Not a valid ESP32 firmware image" toast. The file is correctly rejected. Only the *error message* was suboptimal (fixed as Low priority).

- **Claimed**: "Success path accepts any /api/status response without version-change verification" | **Raised by**: Reviewer A | **Reason**: Design choice, not a bug. The device reboots and becomes unreachable; the first successful `/api/status` response after the reboot window IS the proof of success. Storing the pre-update version for comparison would require extra state management for marginal benefit. Story AC #7 only requires showing the new version string from the response.

- **Claimed**: "Direct Preferences access in WebPortal is an architectural deviation" | **Raised by**: Reviewer B | **Reason**: Explicitly sanctioned in the story's Dev Notes: "Rather than adding this to ConfigManager's formal config structs (overkill for a simple flag), use direct NVS access in WebPortal." This was a deliberate design decision, not a deviation.

- **Claimed**: "No automated test coverage for new OTA/export/rollback behavior" | **Raised by**: Reviewer A | **Reason**: The story scope (Tasks 1-10) does not include test writing for the web UI layer; this project's test suite covers embedded C++ firmware logic in `firmware/test/`. Web UI integration tests are out of scope for this story. The server-side endpoints (`ack-rollback`, `settings/export`) use the same patterns as all other endpoints, which have no dedicated unit tests in this project.

- **Claimed**: "Story file traceability is false (WebPortal.h and main.cpp changed, SystemStatus.cpp didn't)" | **Raised by**: Reviewer A | **Reason**: Documentation gap in the story's "Files NOT to modify" list vs actual diff. Not a code defect. No fix needed in source code.

- **Claimed**: "Upload button not disabled during async FileReader validation" | **Raised by**: Reviewer C | **Reason**: False positive. The `#btn-upload-firmware` button lives inside `#ota-file-info` which has `display:none` during validation. The button is invisible/inaccessible until validation completes successfully.

- **Claimed**: "`aria-valuenow` not set during initial upload state (remains at 0)" | **Raised by**: Reviewer C | **Reason**: False positive. `aria-valuenow="0"` in the HTML is an accurate representation of a not-yet-started upload. Screen readers will correctly announce 0% progress at that state.

- **Claimed**: "Hardcoded `#2d2738` background in `.banner-warning` contradicts CSS variables" | **Raised by**: Reviewer B | **Reason**: AC #9 explicitly specifies "hardcoded `#2d2738` background" as a requirement. This is intentional per the UX specification.
