### OTAUploadState Helper Functions

Following the pattern established by `LogoUploadState` in WebPortal.cpp, implement these helpers:

```cpp
// Find OTA upload state by request pointer
OTAUploadState* findOTAUpload(AsyncWebServerRequest* request) {
    for (auto& state : g_otaUploads) {
        if (state.request == request) {
            return &state;
        }
    }
    return nullptr;
}

// Clear OTA upload state and abort if in progress
void clearOTAUpload(AsyncWebServerRequest* request) {
    for (auto it = g_otaUploads.begin(); it != g_otaUploads.end(); ++it) {
        if (it->request == request) {
            // CRITICAL: Abort update if it was started
            if (it->started) {
                Update.abort();
                Serial.println("[OTA] Upload aborted (cleanup)");
            }
            // Free SHA256 context
            mbedtls_sha256_free(&it->sha256Ctx);
            g_otaUploads.erase(it);
            return;
        }
    }
}
```

**Key implementation notes:**
- `findOTAUpload()` returns nullptr if request not found (check before use)
- `clearOTAUpload()` MUST call `Update.abort()` if `started == true`
- Always call `clearOTAUpload()` in request handler after sending response
- Register `request->onDisconnect()` to call `clearOTAUpload()` for connection failures

### Firmware Integrity Verification (AC7)

**SHA256 Hash Calculation:**
Use mbedtls (already available in ESP32 Arduino core) for incremental hashing:

```cpp
#include <mbedtls/sha256.h>

// In OTAUploadState struct:
mbedtls_sha256_context sha256Ctx;

// On first chunk (index == 0):
mbedtls_sha256_init(&state.sha256Ctx);
mbedtls_sha256_starts_ret(&state.sha256Ctx, 0); // 0 = SHA256 (not SHA224)

// On each chunk (including first):
mbedtls_sha256_update_ret(&state.sha256Ctx, data, len);

// On final chunk:
uint8_t calculatedHash[32];
mbedtls_sha256_finish_ret(&state.sha256Ctx, calculatedHash);

// Compare with X-Firmware-Hash header (hex string)
String expectedHash = request->header("X-Firmware-Hash");
String calculatedHashHex = "";
for (int i = 0; i < 32; i++) {
    char buf[3];
    sprintf(buf, "%02x", calculatedHash[i]);
    calculatedHashHex += buf;
}

if (calculatedHashHex != expectedHash.toLowerCase()) {
    Update.abort();
    state.valid = false;
    state.error = "Firmware integrity check failed";
    state.errorCode = "INTEGRITY_FAILED";
    return;
}
```

**Client Responsibility (Story fn-1.6):**
- Calculate SHA256 hash of firmware binary before upload
- Send hash in `X-Firmware-Hash` header as lowercase hex string (64 characters)
- Example: `X-Firmware-Hash: a3f5b8c9...` (full 64-char hash)

### Firmware Version Validation (AC8)

**Version Comparison Logic:**
```cpp
// Extract version from header (client must provide)
String uploadedVersion = request->header("X-Firmware-Version");
String currentVersion = String(FW_VERSION); // From build flag

// Simple lexicographic comparison (assumes semver format: MAJOR.MINOR.PATCH)
// For production: use proper semver parsing library
if (uploadedVersion < currentVersion) {
    // Check for override flag
    String allowDowngrade = request->header("X-Allow-Downgrade");
    if (allowDowngrade != "true") {
        state.valid = false;
        state.error = "Firmware downgrade not permitted";
        state.errorCode = "DOWNGRADE_BLOCKED";
        Serial.printf("[OTA] Blocked downgrade: %s -> %s\n", 
                      currentVersion.c_str(), uploadedVersion.c_str());
        return;
    }
    Serial.println("[OTA] Downgrade override enabled");
}

Serial.printf("[OTA] Version check passed: %s -> %s\n", 
              currentVersion.c_str(), uploadedVersion.c_str());
```

**Version Extraction Strategy:**
- **Preferred**: Client sends `X-Firmware-Version` header (simplest, no binary parsing)
- **Alternative**: Parse version from binary header (complex, ESP32 binary format specific)
- For fn-1.3: Use header-based approach; binary parsing can be Story fn-1.5 enhancement

**Downgrade Override:**
- Development/recovery scenario: Allow `X-Allow-Downgrade: true` header
- Log all downgrade attempts (even when allowed) for security auditing
- Production UI (fn-1.6) should warn user before allowing downgrade
