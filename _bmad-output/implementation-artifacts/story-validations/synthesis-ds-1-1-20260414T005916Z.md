## Issues Dismissed

- **Claimed: "Missing placement marker in setup()"** | **Raised by:** Validator A (rated Critical) | **Dismissal:** The story's "Heap Baseline Placement" section already enumerates all 9 ordered init calls. Moreover, `main.cpp` lines 806–811 show the placement is already present with a comment. The developer will see both the section and the existing `#if LOG_LEVEL` block. Not a blocking gap.
- **Claimed: "RenderContext lacks const qualifier to enforce NFR S5"** | **Raised by:** Validator A (rated Critical) | **Dismissal:** In C++, you cannot make `FastLED_NeoMatrix*` read-only while still allowing the mode to draw to it — that is the intended contract. `const RenderContext&` prevents copying or overwriting the struct's value fields, which is the correct enforcement point for NFR S5 (brightness field). This is not a fixable design gap; it is the expected C++ behaviour. Added a clarifying sentence to AC #4 instead of treating this as a defect.
- **Claimed: "Heap constant type (uint32_t) unjustified"** | **Raised by:** Validator A | **Dismissal:** The "constant" was redesigned into a documentation comment (no longer a typed constant), so the type question is moot. Even if it were typed, `uint32_t` matching `ESP.getFreeHeap()` return type is self-evident.
- **Claimed: "Relocate HEAP_BASELINE_AFTER_BOOT to ConfigManager.h"** | **Raised by:** Validator B | **Dismissal:** The value is now a documentation comment (not a symbol), so placement in `DisplayMode.h` — the file modes actually include — is the most useful location for future mode authors to reference during sizing. Moving it to `ConfigManager.h` would bury it away from its consumers.
