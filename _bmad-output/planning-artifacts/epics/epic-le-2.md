## Epic le-2: Layout Editor V2 — Richer Fields, Shared Glyphs, Dashboard Live Preview

Follow-on to LE-1 (closed 2026-04-22). This epic captures the editor/rendering work that landed in the working tree after LE-1 shipped and did not fit LE-1's zone. Scope splits into three coherent slices, all layout-editor-adjacent:

1. **Extended FlightField / Metric catalog** — new selectable fields: `aircraft_short`, `aircraft_full`, `origin_iata`, `destination_iata`, and the new metric `speed_mph`. Adds `aircraft_display_name_full` to `FlightInfo` so "Boeing 737-800" can coexist with the marketing "737-800" short form, with fallback precedence mirroring the existing resolvers (`_full` → `_short` → code → "--").
2. **Shared WidgetFont module** — extracts glyph dispatch into `firmware/widgets/WidgetFont.{cpp,h}` so `FlightFieldWidget`, `MetricWidget`, `TextWidget`, and `ClockWidget` render through a common 5×7 / TomThumb / Picopixel selector instead of each reaching into Adafruit_GFX directly.
3. **Dashboard Layout Live Preview** — browser-side pixel-accurate preview of `CustomLayoutMode` rendered into a `<canvas>` inside the dashboard. Mirrors firmware widget render math in JS (`firmware/data-src/preview.js`) against a JS-exported copy of the Adafruit_GFX default, TomThumb, and Picopixel fonts (`preview-glyphs.js`, auto-generated by `tools/generate_preview_glyphs.py`). Backed by a new `/api/flights/current` endpoint and a main-loop `getCurrentFlights()` snapshot of the latest enriched double-buffered flight vector.

**Out of scope:**
- New widget *types* (still the LE-1 set: text, clock, logo, flight_field, metric).
- Editor UX overhaul (the field-picker modal additions are minimal surface-area for the new catalog entries; no larger redesign).
- Server-side rendering of the preview (kept fully client-side — the firmware only publishes flight JSON).

**Dependencies:**
- **LE-1** (closed) — widget registry, editor canvas, field-picker property panel.
- **td-8 aggregator migration** (landing in parallel) — `getCurrentFlights()` reads the aggregator-populated double-buffer; LE-2.3 won't function against the legacy OpenSky/AeroAPI path.

**Coordination with HW-1:** the preview canvas is canvas-size-agnostic (reads `matrix.width/height` from `/api/layout`), so the preview will automatically render at 192×96 once HW-1 bumps the reported canvas size. No LE-2 changes required when HW-1 lands.

### Story le-2.1: Extended FlightField catalog (aircraft-short/full, IATA origin/destination)

As a **layout author**,
I want **`aircraft_short`, `aircraft_full`, `origin_iata`, and `destination_iata` available in the FlightField widget's field picker**,
So that **I can choose between marketing airframe names ("737-800"), full names ("Boeing 737-800"), and IATA airport codes ("JFK") without hand-editing layout JSON**.

**Status:** review

**Acceptance Criteria:**

**Given** the FlightField widget catalog **When** the editor loads `/api/widgets/types` **Then** it advertises a 10-entry FlightField catalog (`callsign`, `airline`, `aircraft_type`, `aircraft_short`, `aircraft_full`, `origin_icao`, `origin_iata`, `destination_icao`, `destination_iata`, `flight_number`); `FlightInfo` exposes `aircraft_display_name_full` populated by the CDN lookup in `FlightDataFetcher`; the `airline` resolver gains a fallback to `operator_icao` when the CDN name hasn't resolved yet, matching the existing aircraft resolver precedence; saved layouts referencing the new IDs pass the save-time whitelist.

---

### Story le-2.2: Extended Metric catalog (speed_mph)

As a **layout author**,
I want **ground speed in mph available alongside the existing knots metric**,
So that **I can build US-unit-oriented layouts without client-side conversion**.

**Status:** review

**Acceptance Criteria:**

**Given** the Metric widget catalog **When** the editor loads `/api/widgets/types` **Then** it advertises a 7-entry metric catalog (`altitude_ft`, `speed_kts`, `speed_mph`, `heading_deg`, `vertical_rate_fpm`, `distance_nm`, `bearing_deg`); `FlightInfo::speed_mph` is populated at fetch time (converted from the upstream feed's m/s); the `speed_mph` resolver passes through without re-converting; NaN rendering falls back to "--" consistent with the other metrics.

---

### Story le-2.3: Shared WidgetFont module

As a **firmware engineer**,
I want **widget glyph rendering consolidated into `firmware/widgets/WidgetFont.{cpp,h}`**,
So that **the five core widgets share a single font-selector code path and widget-level font switching (default / TomThumb / Picopixel) can be added without touching each widget's .cpp**.

**Status:** review

**Acceptance Criteria:**

**Given** the WidgetFont module **When** `FlightFieldWidget`, `MetricWidget`, `TextWidget`, `ClockWidget` render **Then** they dispatch through `WidgetFont::draw()` with a font descriptor rather than calling `display.setFont(...)` directly; the existing golden-frame tests (LE-1.9) still pass byte-for-byte against the default 5×7 font; no per-widget heap-allocating font state is introduced; the module is unit-testable on the host (no `Adafruit_GFX` hardware dependency at the test layer — glyph tables are POD).

---

### Story le-2.4: `/api/flights/current` endpoint and main-loop flight snapshot

As a **dashboard**,
I want **a JSON endpoint that returns the latest enriched flight list**,
So that **the layout live preview can render against real current data**.

**Status:** review

**Acceptance Criteria:**

**Given** `main.cpp` manages a Core 1 double-buffered `std::vector<FlightInfo> g_flightBuf[2]` **When** a caller invokes `getCurrentFlights()` **Then** it returns a copy of the read-side buffer (the one Core 1 is *not* currently writing to); the copy is safe against concurrent writer flips (tens-of-microseconds race window is acceptable for a polling endpoint); `/api/flights/current` serializes the vector to JSON using the standard `{ ok, data, error, code }` envelope; payload size stays under 4 KB for a realistic 8-flight fleet.

---

### Story le-2.5: Dashboard layout live preview

As an **owner**,
I want **a pixel-accurate browser preview of my custom layout painting current flight data**,
So that **I can iterate on widget placement without re-flashing LittleFS between each tweak**.

**Status:** review

**Acceptance Criteria:**

**Given** the dashboard custom-layout card **When** a user opens the live preview with an active layout **Then** `firmware/data-src/preview.js` (backed by `preview-glyphs.js`) renders the widget list into a `<canvas>` at matrix dimensions read from `/api/layout`; flight-bound widgets (`flight_field`, `metric`, `logo`) paint using data fetched from `/api/flights/current`; Adafruit_GFX text advance, line spacing, and TomThumb / Picopixel glyph metrics match the firmware renderer within ±1 px; preview re-renders when the user switches the active layout via the existing activate endpoint.

---

### Story le-2.6: Font extraction tool and regeneration discipline

As a **maintainer**,
I want **`tools/generate_preview_glyphs.py` as the canonical extraction path for preview fonts**,
So that **`preview-glyphs.js` stays in sync with the Adafruit_GFX fonts actually linked into the firmware**.

**Status:** review

**Acceptance Criteria:**

**Given** a fresh checkout **When** `python3 tools/generate_preview_glyphs.py` runs **Then** it reads `glcdfont.c`, `Fonts/TomThumb.h`, and `Fonts/Picopixel.h` from the PlatformIO-vendored Adafruit_GFX library and regenerates `firmware/data-src/preview-glyphs.js` with `GLCD_FONT` (1280-byte Uint8Array), `TOMTHUMB_BITMAPS` / `_GLYPHS`, and `PICOPIXEL_BITMAPS` / `_GLYPHS`; the generated file is byte-identical across runs for the same Adafruit_GFX version; a developer note in the tool's docstring documents when to rerun it (library version bumps).

---

### Story le-2.7: Golden-frame regression + smoke test

As a **maintainer**,
I want **a regression gate that catches font or catalog drift between the preview and the firmware renderer**,
So that **"looks right in preview, wrong on hardware" bugs are caught at CI time**.

**Status:** backlog

**Acceptance Criteria:**

**Given** the LE-1.9 golden-frame corpus extended with one canonical layout exercising each new catalog entry **When** tests run **Then** firmware renders match the golden bytes; a new JS-side smoke script (`tests/smoke/test_preview_renderer.py` or `tools/preview_regression.js`) compares `preview.js` output to a reference image per layout and fails if pixel-diff exceeds a documented tolerance; the `/api/widgets/types` JSON schema and `/api/flights/current` envelope are frozen under a contract test.
